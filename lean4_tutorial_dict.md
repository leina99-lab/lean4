### A-1 콜론 `:`  —  "이것의 종류는 ~이다"

콜론(`:`)은 Lean 4에서 가장 많이 보이는 기호이다.  
**의미: "왼쪽은 오른쪽 타입에 속한다"**

```lean
-- 읽는 법:  이름 : 타입

3 : Nat        -- 3은 자연수(Nat)이다
true : Bool    -- true는 불리언(Bool)이다
h : a ≤ b     -- h는 'a ≤ b'라는 명제의 증명이다
-- 이것을 그대로 실행하면 에러난다. 당황하지 말자. 
```

**비유:** 도서관 책 라벨. 책(왼쪽)에 장르 라벨(오른쪽)이 붙어 있다.
`해리 포터 : 소설`, `1984 : 소설`. 마찬가지로 Lean에서는 `3 : Nat`처럼 데이터에 종류 표시가 붙는다.

> **핵심:** `h : a ≤ b`처럼 명제를 타입으로 쓸 때, `h`는 그 명제의 증명(증거)이다.
> "h라는 이름의 증거가 존재하는데, 그 증거가 증명하는 내용은 `a ≤ b`이다"라는 뜻이다.

### A-2 콜론-등호 `:=`  —  "이것의 내용(값)은 ~이다"

`:=`는 대입(assignment)이다. `:` 뒤에 타입을 쓰고, `:=` 뒤에 실제 값(내용)을 쓴다.

```lean
def x : Nat := 5
--  ↑    ↑       ↑
--  이름  타입    값(내용)

theorem easy : 2 + 2 = 4 := rfl
--       ↑         ↑          ↑
--      이름       명제(타입)   증명(값)
```

**비유:** 주민등록증 양식.
`이름: 홍길동, 생년월일: 1990년` → `이름 : 타입 := 값`
콜론(`:`)은 "이 칸의 종류는", 콜론-등호(`:=`)는 "실제 내용은"이라는 뜻이다.

### A-3 `⊢` 턴스타일  —  "지금 이것을 증명해야 한다"

`⊢` 기호는 "turnstile(턴스타일)"이라고 부른다.
VS Code 오른쪽 **InfoView** 창에서 보인다.

**⊢ 아래에 있는 것이 지금 증명해야 할 목표이다.**

```
-- InfoView 화면 예시:
1 goal
a b : ℝ
h : a ≤ b
⊢ a + 1 ≤ b + 1   ← 이것을 증명해야 한다!
```

| ⊢ 위 (가정/컨텍스트) | ⊢ 아래 (목표/goal) |
|---|---|
| 이미 알고 있는 사실들 | 지금 당장 증명해야 할 것 |
| `a b : ℝ` → a, b는 실수 | `a + 1 ≤ b + 1` → 이것을 보여라 |
| `h : a ≤ b` → a ≤ b라는 증거 h 존재 | 전술을 쓸수록 목표가 바뀐다 |

**비유:** 법정의 칠판.
판사가 칠판에 "오늘 증명해야 할 것: `a + 1 ≤ b + 1`"이라고 써 놓은 것과 같다.
`⊢` 위에 있는 것들은 "이미 알고 있는 사실들(가정)"이다.

> **왜 턴스타일(`⊢`) 기호가 안 보이나요?**
>
> `⊢` 기호는 **'아직 증명해야 할 목표'가 있을 때만** InfoView에 나타난다.
>
> - 증명 중: `⊢ 목표 식` — 이 식을 증명해야 한다는 뜻이다.
> - 증명 완료: `No goals` — 더 이상 증명할 식이 없으므로 기호도 사라진다.
>
> 증명이 잘 되었는지 눈으로 다시 확인하고 싶다면, 커서를 위로 옮겨 확인한다.
> 1. `by` 바로 뒤를 클릭하면 증명 시작 시점의 목표가 다시 나타난다.
> 2. 각 전술 줄 끝을 클릭하면 그 시점의 중간 목표 상태를 볼 수 있다.
> 3. 마지막 줄 끝으로 돌아오면 기호가 사라지며 `No goals`가 표시된다.

### A-4 `→` 오른쪽 화살표  —  "이면" (if ... then)

`→` 기호는 두 가지 상황에서 쓰인다.

| 상황 | 예시 | 읽는 법 |
|---|---|---|
| 함수 타입 | `f : Nat → Nat` | 자연수를 받아서 자연수를 반환하는 함수 |
| 논리적 함의 | `h : P → Q` | P이면 Q이다 |

```lean
-- 경우 1: 함수 타입
def double : Nat → Nat := fun n => n * 2

-- 경우 2: 논리 함의
import Mathlib.Tactic
theorem h (a b : ℝ) : a ≤ b → a + 1 ≤ b + 1 := by
  intro ha   -- 'a ≤ b이다'를 ha라는 이름으로 가정에 추가
  linarith   -- 자동 증명
```

**비유:** "비가 오면 → 땅이 젖는다."
비(왼쪽)가 실제로 왔을 때만 땅이 젖는다(오른쪽)는 것을 보장한다.
비가 안 왔을 때 땅이 어떤지는 말하지 않는다. 이것이 `→`의 한계이자 특징이다.

> **주의:** `→`는 한 방향이다. `A → B`가 참이어도 `B → A`가 참이라는 뜻이 아니다.
> 양 방향이 모두 필요하면 `↔`(A-5)를 써야 한다.

---

### A-5 `↔` 양방향 화살표  —  "일 때 그리고 그때만" (iff)

**↔ = iff = if and only if = "~이면 그리고 그때만"**

`A ↔ B`는 `A → B`이고 동시에 `B → A`이다. 두 방향이 모두 성립한다.

```lean
-- exp_le_exp 라는 정리:
-- exp_le_exp : exp a ≤ exp b  ↔  a ≤ b
--               ↑ 왼쪽(정방향)       ↑ 오른쪽(역방향)

-- .mp  : 정방향 (→ 방향)  exp a ≤ exp b → a ≤ b
-- .mpr : 역방향 (← 방향)  a ≤ b → exp a ≤ exp b

example (h : a ≤ b) : exp a ≤ exp b :=
  exp_le_exp.mpr h   -- 역방향(.mpr)에 h를 넣어 증명 끝
```

| ↔의 두 방향 | 이름 | 의미 |
|---|---|---|
| `h.mp` | 정방향 | A → B 방향 |
| `h.mpr` | 역방향 | B → A 방향 |
| `h.1` | h.mp와 동일 | 첫 번째 방향 |
| `h.2` | h.mpr과 동일 | 두 번째 방향 |

**비유:** "삼각형이 정삼각형이다 ↔ 세 변의 길이가 같다."
정삼각형이면 세 변이 같고(→), 세 변이 같으면 정삼각형이다(←).
양방향이 모두 성립하기 때문에 `↔`를 쓸 수 있다.

### A-6 `∧` 논리 곱  —  "그리고" (and)

`A ∧ B`는 "A이고 B이다"라는 뜻이다. A와 B 모두 참이어야 전체가 참이다.

```lean
-- h : P ∧ Q  →  'P이고 Q이다'라는 증거 h가 있다
example (P Q : Prop) (h : P ∧ Q) : P := h.1  -- P의 증거
example (P Q : Prop) (h : P ∧ Q) : Q := h.2  -- Q의 증거

-- 또는 증명 블록 안에서:
example (P Q : Prop) (h : P ∧ Q) : P ∧ Q := by
  constructor
-- 꺼내기:
  · exact h.1  -- P의 증거
  · exact h.2  -- Q의 증거

-- 만들기:
example : 2 > 1 ∧ 3 > 2 := by
  constructor     -- 두 개의 목표로 분리
  · norm_num      -- 2 > 1 증명
  · norm_num      -- 3 > 2 증명
```

**비유:** AND 게이트. 두 스위치가 모두 켜져야(true) 전구가 켜진다.

---

### A-7 `∨` 논리 합  —  "또는" (or)

`A ∨ B`는 "A이거나 B이다"라는 뜻이다. 둘 중 하나만 참이어도 전체가 참이다.

```lean
-- 만들기:  Or.inl (왼쪽), Or.inr (오른쪽)
example : 2 = 2 ∨ 3 = 4 := Or.inl rfl   -- 왼쪽(2=2)이 참이므로
example : 1 = 2 ∨ 3 = 3 := Or.inr rfl   -- 오른쪽(3=3)이 참이므로
```

---

### A-8 `¬` 부정  —  "아니다" (not)

`¬P`는 "P가 아니다"라는 뜻이다. Lean에서 `¬P`는 사실 `P → False`와 같다.

```lean
example : ¬ (2 = 3) := by
  intro h    -- h : 2 = 3 이라고 가정
  exact absurd h (by decide)   -- 모순 도출
```

---

### A-9 `∀` 전칭 한정자  —  "모든 ~에 대해" (for all)

`∀ x, P x`는 "모든 x에 대해 P(x)가 성립한다"라는 뜻이다.

```lean
theorem zero_add : ∀ x : Nat, x + 0 = x := by
  intro n    -- 임의의 n을 하나 가져온다
  rfl        -- n + 0 = n은 정의상 참

-- 사용하기: 특정 값을 넣으면 됨
#check zero_add 5   -- 5 + 0 = 5 : Prop
```

**비유:** "모든 백조는 하얗다 (∀ x, 백조(x) → 하얗다(x))"
`intro n`은 "임의의 백조 한 마리(n)를 골라오세요"라는 전술이다.

---

### A-10 `∃` 존재 한정자  —  "~가 존재한다" (exists)

`∃ x, P x`는 "P(x)를 만족하는 x가 적어도 하나 존재한다"라는 뜻이다.

```lean
-- '3보다 큰 자연수가 존재한다'
example : ∃ n : Nat, n > 3 := by
  use 4      -- 4를 증거로 제시
  norm_num   -- 4 > 3 자동 증명
```

### A-11 수 체계 기호 `ℕ ℤ ℝ ℚ`  —  숫자의 종류

| 기호 | 이름 | 포함하는 수 | Lean 이름 | 비고 |
|---|---|---|---|---|
| `ℕ` | 자연수 | 0, 1, 2, 3, ... | `Nat` | 음수 없음! 3-5=0 |
| `ℤ` | 정수 | ...-2,-1,0,1,2,... | `Int` | 음수 있음 |
| `ℚ` | 유리수 | 분수 포함 | `Rat` | a/b 형태 |
| `ℝ` | 실수 | √2, π 포함 | `Real` | 가장 많이 씀 |

> **주의:** `ℕ`에서는 음수가 없다! `(3 : Nat) - 5`를 계산하면 `-2`가 아니라 `0`이 나온다.
> 이것을 "포화 뺄셈(truncated subtraction)"이라 한다.

### A-12 `←` 역방향 화살표  —  rw에서의 역방향 치환

`←`는 `rw`(치환) 전술 안에서만 쓰인다. `rw [← h]`는 h를 거꾸로 적용하라는 뜻이다.

```lean
-- mul_comm a b  :  a * b = b * a

rw [mul_comm a b]    -- 목표에서 'a*b'를 'b*a'로 바꿈  (정방향)
rw [← mul_comm a b]  -- 목표에서 'b*a'를 'a*b'로 바꿈  (역방향)

-- ← 입력 방법 (VS Code): \l  또는  \leftarrow  입력 후 탭/엔터
```

### A-13 `·` 가운뎃점  —  "이 목표에 집중한다"

`constructor`, `cases` 같은 전술로 목표가 여러 개로 쪼개질 때,
각 목표에 대한 증명을 `·`로 구분한다.

```lean
example : 2 > 1 ∧ 3 > 2 := by
  constructor      -- 목표가 2개로 쪼개짐
  · norm_num       -- 첫 번째 목표(2 > 1) 증명
  · norm_num       -- 두 번째 목표(3 > 2) 증명
```

가운뎃점 `·`은 "이 목표에 집중합니다"를 뜻한다. 목표가 여러 개일 때 각각 구분해서 증명하는 데 쓴다.

### A-14 `_` 언더스코어  —  "이전 줄의 결과" 또는 "신경 안 씀"

```lean
-- 용도 1: calc 블록에서 '이전 줄의 결과'
calc
  a + b + c
    = a + (b + c) := by ring
  _ = a + (c + b) := by rw [add_comm b c]   -- _ = 이전 줄 결과

-- 용도 2: 이름이 필요 없는 인수
fun _ => 42    -- 입력값은 신경 안 쓰고 항상 42를 반환
```

### B-1 `#check`와 `#eval`  —  "물어보는" 명령어

`#check`와 `#eval`은 Lean 4에서 정보를 물어보는 명령어이다.
이 명령어들은 증명에 포함되지 않으며, 학습과 디버깅에 쓴다.

```lean
-- #check: '이것의 타입이 뭐야?'
#check 2 + 2          -- 2 + 2 : Nat
#check (2 + 2 : ℝ)   -- 2 + 2 : ℝ
#check mul_comm       -- mul_comm : ∀ (a b : ?α), a * b = b * a

-- #eval: '계산해서 결과 보여줘'
#eval 2 + 2            -- 4
#eval [1,2,3].length   -- 3
#eval String.length "hello"  -- 5
```

| 명령어 | 용도 | 결과 |
|---|---|---|
| `#check X` | X의 타입을 물어봄 | `X : 타입` 형태로 출력 |
| `#eval X` | X를 실제로 계산함 | 계산 결과 출력 |
| `#print X` | X의 정의를 출력 | 정의 전체 출력 |

### B-2 `def`  —  "이름을 붙여 정의한다"

`def`는 definition(정의)의 줄임말이다. 함수나 값에 이름을 붙일 때 쓴다.

```lean
-- 기본 형태:  def 이름 : 타입 := 값

def myNum : Nat := 42
def double (n : Nat) : Nat := n * 2
def greet (name : String) : String := "Hello, " ++ name

#eval double 5       -- 10
#eval greet "Alice"  -- "Hello, Alice"
```

### B-3 `theorem` / `lemma` / `example`  —  수학 명제 선언

> **핵심:** Lean 4에서 `theorem`과 `lemma`는 기능적으로 완전히 동일하다.
> 차이는 "읽는 사람을 위한 힌트"뿐이다.

```lean
-- theorem: '이것은 중요한 메인 결과입니다'
theorem pythagorean : a^2 + b^2 = c^2 := sorry

-- lemma: '이것은 더 큰 증명을 위한 보조 도구입니다'
lemma helper_lemma : a * 0 = 0 := by ring

-- example: '이름 없이 연습합니다'
example : 2 + 2 = 4 := rfl
```

| 키워드 | 이름 저장 | 재사용 | 인간에게 보내는 신호 |
|---|---|---|---|
| `theorem` | 저장됨 | 가능 | "중요한 최종 결과" |
| `lemma` | 저장됨 | 가능 | "보조 도구, 중간 단계" |
| `example` | 없음 | 불가능 | "연습용, 데모용" |

**비유:** 요리 레시피.
`theorem` = 완성된 메뉴(메인 요리),
`lemma` = 재료 손질 단계(소스 만들기 등),
`example` = 시식 연습(손님에게 내기 전 맛 보기).

### B-4 `by`  —  "지금부터 전술 모드로 증명하겠다"

`by`는 "이제부터 전술(tactic)을 써서 단계적으로 증명하겠다"는 선언이다.

```lean
-- 항 모드 (term mode): 증명을 직접 하나의 값으로 씀
theorem t1 : 2 + 2 = 4 := rfl

-- 전술 모드 (tactic mode): by 이후에 전술들을 단계별로 씀
theorem t2 : 2 + 2 = 4 := by
  rfl

-- 복잡한 증명은 전술 모드가 편하다
theorem t3 (a b : ℝ) (h : a ≤ b) : a ≤ b + 1 := by
  linarith
```

**비유:** `by` = "단계별 요리 시작".
항 모드는 "완성된 음식을 바로 제출",
전술 모드는 "1단계: 재료 준비, 2단계: 볶기, 3단계: 플레이팅"처럼 단계별로 만드는 것.

---

### B-5 `rfl`  —  "양쪽이 정의상 같다"

`rfl`은 reflexivity(반사성)의 줄임말이다.
"양쪽을 계산하면 정확히 같은 값이 된다"는 것을 증명한다.

```lean
-- rfl이 되는 경우: 계산하면 양쪽이 같아짐
theorem ex1 : 2 + 2 = 4 := rfl
theorem ex2 : [1,2] ++ [3] = [1,2,3] := rfl

-- rfl이 안 되는 경우: 변수가 있어서 계산 불가
-- theorem ex3 (a b : Nat) : a + b = b + a := rfl  -- 오류!
-- a, b가 뭔지 모르니 계산 불가 → ring이나 omega를 써야 함
```

```
-- InfoView 변화:
-- rfl 전:   ⊢ 2 + 2 = 4
-- rfl 후:   goals accomplished  (증명 완료!)
```

**비유:** 저울에 양쪽을 올려놓고 계산하면 무게가 같다는 것을 확인.
Lean이 양쪽을 직접 계산해서 "응, 같네!" 하는 순간이 `rfl`이다.

---

### B-6 `rw`  —  치환(대입/슈퍼포지션): "같은 것을 바꿔치기"

`rw`는 rewrite(다시 쓰기)의 줄임말이다.
"A = B"라는 등식을 이용해 목표에서 A를 B로 바꾼다.

```lean
rw [h]           -- h : A = B  →  목표에서 A를 B로 바꿈
rw [← h]         -- h : A = B  →  목표에서 B를 A로 바꿈 (역방향)
rw [h] at hyp    -- hyp라는 가정에서 치환
rw [h1, h2, h3]  -- 연속 치환 (한 줄에)

-- 실제 예시
example (a b c : ℝ) : a * b * c = b * (a * c) := by
  rw [mul_comm a b]    -- a*b → b*a
  rw [mul_assoc b a c] -- b*a*c → b*(a*c)
```

```
-- InfoView 변화:
-- rw [mul_comm a b] 전:   ⊢ a * b * c = b * (a * c)
-- rw [mul_comm a b] 후:   ⊢ b * a * c = b * (a * c)
--                              ↑ a와 b가 바뀌었다!
```

위 예시의 두 단계를 따라가면 다음과 같다.

**1단계**: `rw [mul_comm a b]` 실행 후 목표: `⊢ b * a * c = b * (a * c)`

**2단계**: `rw [mul_assoc b a c]` 실행 후 목표: `⊢ b * (a * c) = b * (a * c)`

양쪽이 같으므로 Lean이 자동으로 증명을 완료한다.

> **왜 마지막에 `⊢` 기호가 사라지나요?**
>
> `⊢` 기호는 **'아직 증명해야 할 목표'가 있을 때만** 나타난다. 증명이 완료되면 `No goals`가 표시되고 기호도 함께 사라진다.
>
> 증명 중간 상태를 다시 확인하고 싶다면 커서를 해당 줄로 옮기면 된다.
> 1. `by` 바로 뒤를 클릭하면 `⊢ a * b * c = b * (a * c)`가 다시 나타난다.
> 2. 첫 번째 `rw` 줄 끝을 클릭하면 `⊢ b * a * c = b * (a * c)`가 보인다.
> 3. 마지막 줄 끝으로 돌아오면 기호가 사라지며 `No goals`가 표시된다.

**비유 (슈퍼포지션):**
"A = B"를 아는 상태에서, 종이 위의 "A"라는 글자 위에 "B"라는 스티커를 붙인다.
원래 글자(A) 대신 스티커(B)가 보인다. `rw`는 이 과정을 Lean 목표에서 수행한다.

> **주의:** `rw`에 넣는 `h`는 반드시 `A = B` 또는 `A ↔ B` 형태여야 한다.
> `A → B` 형태는 `rw`에 쓸 수 없다!

#### `rw` 대신 `ring` 한 줄로 끝내기

`rw`를 하나씩 쓰는 방식은 원리를 이해하는 데 매우 좋다. 그러나 Lean 4에는 대수적인 등식을 **한 번에 자동으로 해결**해 주는 `ring`이라는 강력한 전술이 있다.

```lean
import Mathlib.Tactic

example (a b c : ℝ) : a * b * c = b * (a * c) := by
  ring
```

`ring`은 **환(Ring)** 구조에서의 등식 — 교환법칙, 결합법칙, 분배법칙 등 — 을 자동으로 계산하여 증명을 완료한다. 위 코드를 실행하면 InfoView에 `No goals`가 표시된다.

| 방법 | 코드 줄 수 | 언제 쓰나 |
|------|-----------|----------|
| `rw` 단계별 | 여러 줄 | 각 치환 단계를 직접 이해하고 싶을 때 |
| `ring` 자동 | 1줄 | 대수 등식임이 분명할 때 (교환·결합·분배법칙 조합) |

> **주의**: `ring`은 등식(`=`)만 해결한다. 부등식(`≤`, `<`)이나 논리 명제에는 `linarith`, `omega` 등 다른 전술을 써야 한다.

---

### B-7 `exact`  —  "이것이 바로 답이다"

`exact h`는 "현재 목표와 h가 정확히 같다. h가 바로 증명이다"라는 뜻이다.

```lean
example (a b : ℝ) (h : a ≤ b) : a ≤ b := by
  exact h    -- 목표(a ≤ b)와 h(a ≤ b)가 정확히 같음 → 끝!
```

**비유:** 퍼즐 조각을 맞추는 것.
목표(구멍 모양)와 내가 가진 조각(h)의 모양이 정확히 같으면 `exact h`로 끼워 넣는다.

---

### B-8 `intro`  —  "~라고 가정하자"

목표가 `A → B` 형태일 때, `intro h`는 "A가 참이라고 가정하고 h라는 이름을 붙인다"는 뜻이다.
그러면 목표가 B로 바뀐다.

```lean
example (a b : ℝ) : a ≤ b → a + 1 ≤ b + 1 := by
  -- 초기 상태:  ⊢ a ≤ b → a + 1 ≤ b + 1
  intro h
  -- intro h 후:  h : a ≤ b  /  ⊢ a + 1 ≤ b + 1
  linarith

-- ∀ 에서도 intro 사용
example : ∀ n : Nat, n + 0 = n := by
  intro n     -- 임의의 n 하나를 가져옴
  rfl
```

**비유:** "비가 온다고 치자. 그러면..."
`intro h`는 전제("비가 온다")를 h라는 이름으로 가정 목록에 추가하고,
결론("땅이 젖는다")만 증명하면 되는 상태로 만든다.

---

### B-9 `apply`  —  "이 정리를 적용한다"

`apply h`는 정리 h의 결론이 현재 목표와 같을 때,
목표를 h의 전제 조건으로 바꾼다.

```lean
-- add_le_add : a ≤ b → c ≤ d → a + c ≤ b + d

example (a b c d : ℝ) (h1 : a ≤ b) (h2 : c ≤ d) : a + c ≤ b + d := by
  apply add_le_add  -- 목표가 두 개의 전제로 쪼개짐
  · exact h1        -- 목표 1: ⊢ a ≤ b
  · exact h2        -- 목표 2: ⊢ c ≤ d
```

**비유:** "이 정리를 쓸 수 있을까? 그러려면 A, B 두 가지가 필요하다."
`apply` 후에 A, B를 각각 증명하면 된다.

---

### B-10 `constructor`  —  "두 부분을 각각 증명한다"

목표가 `A ∧ B` 또는 `A ↔ B` 형태일 때,
`constructor`는 두 개의 목표로 쪼갠다.

```lean
-- A ∧ B 증명
example : 2 > 1 ∧ 3 > 2 := by
  constructor
  · norm_num
  · norm_num

-- A ↔ B 증명
example (a b : ℝ) : a + b = b + a ↔ True := by
  constructor
  · intro _; trivial
  · intro _; ring
```

---

### B-11 `cases`  —  "경우를 나눠서 살펴본다"

`cases h`는 h의 형태에 따라 경우를 나눠서 각 경우를 따로 증명한다.

```lean
example (P Q R : Prop) (h : P ∨ Q) (hp : P → R) (hq : Q → R) : R := by
  cases h with
  | inl hp' => exact hp hp'   -- P인 경우
  | inr hq' => exact hq hq'   -- Q인 경우
```

---

### B-12 자동 증명 전술  —  ring / linarith / norm_num / omega / decide

이 전술들은 특정 종류의 목표를 자동으로 해결해 주는 편리한 명령어들이다.

| 전술 | 해결하는 것 | 예시 |
|---|---|---|
| `ring` | 환(ring)의 등식 (실수/정수 사칙연산 등식) | `(a+b)^2 = a^2+2*a*b+b^2` |
| `linarith` | 선형 부등식 (≤, <, ≥, > 포함 가설) | `a ≤ b, b ≤ c → a ≤ c` |
| `norm_num` | 구체적 수치 계산 | `2^10 = 1024` |
| `omega` | 자연수/정수 선형 산술 | `n % 2 = 0 → (n+2) % 2 = 0` |
| `decide` | 유한하게 계산 가능한 명제 | `2 < 5` |
| `simp` | 자동 단순화 (기본 규칙 모음 적용) | `a + 0 = a` |

```lean
example (a b : ℝ) : (a + b)^2 = a^2 + 2*a*b + b^2 := by ring
example (a b c : ℝ) (h1 : a ≤ b) (h2 : b ≤ c) : a ≤ c := by linarith
example : (2 : ℝ)^10 = 1024 := by norm_num
example : 2 < 5 := by decide
```

---

### B-13 `sorry`  —  "나중에 채울 미완성 표시"

`sorry`는 증명을 아직 완성하지 못했을 때 임시로 쓰는 플레이스홀더이다.
Lean이 이 부분은 눈감아주는 대신, 경고(warning)를 표시한다.

```lean
-- sorry를 쓰면 Lean이 일단 통과시켜 준다
theorem my_theorem : 어려운_명제 := by
  sorry   -- 노란 경고: 'declaration uses sorry'

-- 연습 문제에서 채워야 할 곳에 씀
example (a b : ℝ) (h : a ≤ b) : exp a ≤ exp b := by
  sorry   -- 여기를 채워야 함
```

> **주의:** `sorry`를 포함한 코드는 최종 제출에서는 반드시 제거해야 한다.
> `sorry`가 있으면 그 정리를 신뢰할 수 없다.

---

### B-14 `calc`  —  "단계별 계산 (풀이 과정 쓰기)"

`calc`는 calculation(계산)의 줄임말이다.
수학 시험에서 풀이 과정을 쓰듯이, 단계별로 변환 과정을 나열한다.

```lean
example (a b : ℝ) : (a + b)^2 = a^2 + 2*a*b + b^2 := by
  calc (a + b)^2
      = (a + b) * (a + b)       := by ring
    _ = a*a + a*b + b*a + b*b   := by ring
    _ = a^2 + 2*a*b + b^2       := by ring
-- _ 는 '이전 줄의 결과'를 가리킴
```

### InfoView 화면 구조

```
2 goals                    ← 아직 증명할 목표가 2개 남았다

case left                  ← 목표의 이름 (constructor로 쪼갤 때)
a b : ℝ                    ← 가정: a, b는 실수이다
h : a ≤ b                  ← 가정: a ≤ b라는 증거 h가 있다
⊢ a + 1 ≤ b + 1            ← 목표: 이것을 증명해야 한다

case right                 ← 두 번째 목표
a b : ℝ
⊢ b - 1 ≤ a → False
```

| InfoView 요소 | 의미 |
|---|---|
| `N goals` | 남은 목표 개수 |
| `이름 : 타입` (⊢ 위) | 가정, 이미 알고 있는 것 |
| `⊢ 명제` (⊢ 아래) | 현재 증명해야 할 목표 |
| `goals accomplished` | 증명 완료 |

## 파트 D | 자주 보이는 오류 메시지

| 오류 메시지 | 의미 | 해결 방법 |
|---|---|---|
| `unknown identifier 'X'` | X라는 이름을 못 찾겠다 | 오타 확인, `#check X`로 이름 확인 |
| `type mismatch` | 타입이 안 맞는다 | `:`로 타입 명시, 가정 확인 |
| `unsolved goals` | 증명이 끝나지 않았다 | 빠진 전술 추가, `sorry`로 임시 처리 |
| `declaration uses 'sorry'` | sorry가 포함됨 (경고) | sorry 자리를 실제 증명으로 교체 |
