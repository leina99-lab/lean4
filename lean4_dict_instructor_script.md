### 8~22분 — A-1 콜론 `:` — 타입 라벨

#### 이 기호가 하는 일

> "Lean 4 코드에서 가장 많이 등장하는 기호가 무엇인지 아세요? 바로 이 콜론(:)입니다. 코드 한 줄에 두세 번씩 나오기도 합니다."
>
> "콜론의 의미는 딱 하나입니다. '왼쪽은 오른쪽 타입에 속한다.' 다시 말하면, '왼쪽 것의 종류가 오른쪽이다'입니다."

```lean
-- 읽는 법: 이름 : 타입

3 : Nat        -- 3은 자연수(Nat)이다
true : Bool    -- true는 불리언(Bool)이다
h : a ≤ b      -- h는 'a ≤ b'라는 명제의 증명이다

-- 위 세 줄을 그대로 실행하면 에러가 난다. 당황하지 말자.
-- 독립적인 선언이 아니라 증명 문맥 안에서 쓰는 표기이기 때문이다.
```

> "도서관 책 라벨을 상상해 보세요. 책 표지에 '해리 포터 : 소설', '코스모스 : 과학'처럼 장르 라벨이 붙어 있죠. Lean에서는 모든 값 옆에 그 값의 장르, 즉 타입이 붙습니다. 3 옆에는 Nat, true 옆에는 Bool."

**Q: 그런데 세 번째 줄이 좀 다릅니다. `h : a ≤ b`. 이건 무슨 뜻일까요?**

*(학생들이 생각할 시간을 준다.)*

> "h는 'a ≤ b라는 명제의 증명'입니다. 이게 Lean 4에서 가장 중요한 철학이에요. Lean에서는 명제도 타입입니다. 그리고 그 타입의 값은 증명입니다."
>
> "h가 존재한다는 것 자체가 'a ≤ b는 참이다'를 의미합니다. 증명을 하나 만들어냈다는 것은 그 명제 타입의 값을 하나 만들어냈다는 것과 같습니다. 이 생각을 오늘 하루 내내 마음에 품고 계세요. 나중에 갈수록 이게 얼마나 강력한 아이디어인지 느끼게 됩니다."

> **[ 시연 ]**
> `#check 3`을 입력하고 InfoView에서 `3 : Nat`이 나오는 것을 보여준다.
> `#check (3 : ℝ)`를 입력하면 같은 3이라도 타입이 ℝ로 바뀌는 것을 보여준다.
> 타입이 맥락에 따라 다를 수 있음을 느끼게 한다.

### 22~35분 — A-2 콜론-등호 `:=` — 명함과 본체

#### 콜론과 콜론-등호의 관계

> "콜론을 배웠으니 이제 콜론-등호(:=)입니다. 이 두 기호는 항상 함께 나타납니다."
>
> "콜론(:)이 '이 칸의 종류는 무엇이다'라면, 콜론-등호(:=)는 '실제 내용은 무엇이다'입니다."
>
> "주민등록증을 떠올려 보세요."

```
이름 : 홍길동
생년월일 : 1990년 1월 1일
직업 : 개발자
```

> "여기서 '이름:' 부분이 콜론의 역할이고, 그 뒤의 '홍길동'이 콜론-등호(:=) 자리에 오는 내용입니다. Lean 코드로 옮기면 이렇게 됩니다."

```lean
def x : Nat := 5
--  ↑    ↑       ↑
-- 이름  타입    값(내용)

theorem easy : 2 + 2 = 4 := rfl
--       ↑         ↑          ↑
--      이름    명제(타입)   증명(값)
```

> "theorem에서는 콜론-등호 오른쪽이 '증명의 실체'가 됩니다. 이 경우 rfl이라는 한 단어가 증명 전체입니다. 더 복잡한 증명이라면 by 키워드 뒤에 수십 줄이 붙기도 합니다. 그 모든 것이 콜론-등호 오른쪽입니다."
>
> "기억하세요. ':' 오른쪽에는 항상 타입(종류), ':=' 오른쪽에는 항상 실제 값(내용)이 옵니다."

### 35~52분 — A-3 턴스타일 `⊢` — 법정의 칠판

#### 가장 중요한 기호

> "자, 이제 여러분이 Lean 4를 사용하면서 가장 자주 마주치는 기호입니다. VS Code 오른쪽 InfoView 창을 열면 보이는 이 누운 T 모양, ⊢. 이름은 '턴스타일(turnstile)'입니다."
>
> "이 기호의 역할은 딱 하나입니다. ⊢ 아래에 있는 것이 지금 증명해야 할 목표입니다."
>
> "InfoView 화면을 직접 보겠습니다."

```
[ InfoView 화면 예시 ]

1 goal
a b : ℝ
h : a ≤ b
⊢ a + 1 ≤ b + 1   ← 이것을 증명해야 한다!
```

> "이걸 한 줄씩 해석해 드리겠습니다."
>
> "첫 줄 '1 goal'은 지금 증명해야 할 목표가 1개 남았다는 뜻입니다."
>
> "'a b : ℝ'은 a와 b가 실수라는 사실이 이미 주어져 있다는 뜻입니다."
>
> "'h : a ≤ b'는 a ≤ b가 참임을 보장하는 증거 h가 존재한다는 뜻입니다."
>
> "그리고 ⊢ 아래의 'a + 1 ≤ b + 1'이 바로 우리가 지금 증명해야 할 목표입니다."
>
> "⊢ 위에 있는 것들은 재료, ⊢ 아래에 있는 것은 목표. 이 구분이 Lean 4 증명을 읽는 핵심입니다."

#### 법정 비유

> "법정의 칠판을 상상해 보세요. 판사가 칠판 중간에 선을 하나 긋고 위에는 이미 제출된 증거들, 아래에는 '오늘 이 법정에서 입증해야 할 사실'을 써놓았습니다. 바로 그 구분선이 ⊢입니다."
>
> "여러분은 변호사입니다. ⊢ 위의 재료들을 활용해서 ⊢ 아래의 목표를 무너뜨려야 합니다. 전술(tactic) 하나를 쓸 때마다 칠판의 내용이 바뀝니다. 목표가 더 단순해지거나, 목표가 여러 개로 쪼개지거나, 혹은 완전히 사라집니다."

#### ⊢가 사라졌을 때

> "가끔 학생들이 '선생님, InfoView에서 ⊢가 사라졌어요!'라고 당황하면서 찾아옵니다. 그럴 때 저는 이렇게 대답합니다. 박수 치세요! 증명이 완료된 겁니다."
>
> "⊢는 아직 증명해야 할 목표가 있을 때만 나타납니다. `No goals`가 뜨면 더 이상 증명할 것이 없다는 뜻입니다. 완전한 승리입니다."

```
[ 증명 완료 상태 ]

No goals

-- 또는
goals accomplished
```

> "중간 과정을 다시 보고 싶다면, VS Code에서 마우스 커서를 코드 위로 이동하면 됩니다."
>
> "`by` 바로 뒤에 커서를 놓으면 증명 시작 직전의 목표가 나타납니다."
>
> "각 전술 줄 끝에 커서를 놓으면 그 전술을 실행한 직후의 목표 상태를 볼 수 있습니다."
>
> "마지막 전술 줄 끝으로 오면 `No goals`가 다시 나타납니다."
>
> "이 커서 이동이 Lean 4 학습에서 가장 강력한 도구입니다. 탐정이 되어 증거를 하나씩 따라가듯이 증명의 흐름을 추적할 수 있습니다."

> **[ 시연 — 반드시 직접 보여줄 것 ]**
>
> VS Code에서 다음 코드를 입력한다.
> ```lean
> example (a b : ℝ) (h : a ≤ b) : a + 1 ≤ b + 1 := by
>   linarith
> ```
> `by` 뒤를 클릭 → InfoView에 초기 목표 표시  
> `linarith` 줄 끝 클릭 → `No goals` 표시  
> 커서를 다시 `by` 뒤로 이동 → 초기 목표 복귀 확인

### 52~60분 — A-4 `→` — 함수이자 논리의 다리

#### 두 가지 얼굴

> "→ 기호는 Lean 4에서 두 가지 완전히 다른 상황에서 등장합니다. 그런데 둘 다 알아야 코드를 읽을 수 있습니다."

```lean
-- 얼굴 1: 함수 타입
def double : Nat → Nat := fun n => n * 2
-- 'Nat → Nat'은 자연수를 받아서 자연수를 돌려주는 함수의 타입이다.

-- 얼굴 2: 논리적 함의 (a, b 타입 선언 필수!)
theorem step_up (a b : ℝ) : a ≤ b → a + 1 ≤ b + 1 := by
  intro ha   -- 'a ≤ b이다'를 ha라는 이름으로 가정에 추가
  linarith   -- 선형 산술 자동 증명
```

> "입장권 비유를 써보겠습니다. '비가 오면 → 땅이 젖는다.' 비라는 구역에 들어오려면 '비가 온다'는 입장권을 내야 합니다. 입장권이 없으면 결론이 성립하는지 아무 말도 할 수 없습니다."
>
> "핵심은 → 가 한 방향이라는 점입니다. A → B가 참이어도 B → A가 참인 건 아닙니다. '비가 오면 땅이 젖는다'가 참이어도, '땅이 젖었으면 비가 온 것이다'는 거짓일 수 있어요. 스프링클러 때문에 젖을 수도 있으니까요."

> **[ 주의 ]**
>
> A-4의 코드 예시에서 `theorem h : a ≤ b → ...`처럼 변수 선언 없이 쓰면 에러가 난다.  
> 반드시 `theorem h (a b : ℝ) : a ≤ b → ...`처럼 `(a b : ℝ)`을 명시해야 한다.

## 1교시 후반 — A-5 `↔`, A-6 `∧`, A-7 `∨` (나머지 20분)

### A-5 `↔` — 양방향 다리

> "→가 한쪽 방향 다리라면 ↔는 양쪽이 모두 열린 다리입니다. A ↔ B는 A → B이고 동시에 B → A인 경우를 말합니다."
>
> "수학에서 가장 강력한 문장 중 하나가 ↔입니다. 두 조건이 완전히 동등하다는 뜻이기 때문이죠."
>
> "예를 들어 '삼각형이 정삼각형이다 ↔ 세 변의 길이가 같다.' 정삼각형이면 세 변이 같고(→), 세 변이 같으면 정삼각형이다(←). 이 두 방향이 모두 성립하기 때문에 ↔를 쓸 수 있습니다."

```lean
import Mathlib.Tactic
open Real

-- Real.exp_le_exp : exp a ≤ exp b ↔ a ≤ b
-- 이 정리는 양방향이므로 두 방향을 각각 꺼낼 수 있다.

-- 정방향 (.mp): exp a ≤ exp b → a ≤ b
example (a b : ℝ) (h : exp a ≤ exp b) : a ≤ b :=
  Real.exp_le_exp.mp h

-- 역방향 (.mpr): a ≤ b → exp a ≤ exp b
example (a b : ℝ) (h : a ≤ b) : exp a ≤ exp b :=
  Real.exp_le_exp.mpr h
```

> "mp는 **modus ponens**(긍정 논법)의 줄임말입니다. 논리학 교과서에 나오는 용어인데, 쉽게 말하면 정방향입니다. mpr은 modus ponens reverse, 즉 역방향입니다."
>
> "`h.1`은 `h.mp`와 같고, `h.2`는 `h.mpr`과 같습니다. 둘 다 쓸 수 있습니다."

### A-6 `∧` — '그리고', 둘 다 참

> "∧는 and입니다. A ∧ B는 'A이고 B이다'입니다. A와 B가 둘 다 참이어야 전체가 참입니다."
>
> "전기 회로의 AND 게이트와 같습니다. 두 스위치가 모두 켜져야만 전구가 켜집니다. 하나라도 꺼지면 전구도 꺼집니다."
>
> "`h : P ∧ Q`가 있을 때, P만 꺼내고 싶으면 `h.1`을, Q만 꺼내고 싶으면 `h.2`를 씁니다. 반대로 P ∧ Q를 만들어야 할 때는 `constructor`로 두 개의 목표로 쪼갠 뒤 각각 증명합니다."

```lean
-- ∧ 꺼내기
example (P Q : Prop) (h : P ∧ Q) : P := h.1   -- P만 꺼낸다
example (P Q : Prop) (h : P ∧ Q) : Q := h.2   -- Q만 꺼낸다

-- ∧ 만들기
example : 2 > 1 ∧ 3 > 2 := by
  constructor     -- 목표를 두 개로 쪼갠다
  · norm_num      -- 첫 번째 목표: 2 > 1 증명
  · norm_num      -- 두 번째 목표: 3 > 2 증명
```

### A-7 `∨` — '또는', 하나만 참이어도 됨

> "∨는 or입니다. A ∨ B는 'A이거나 B이다'입니다. 둘 중 하나만 참이어도 전체가 참입니다."
>
> "∨를 증명할 때는 내가 어느 쪽을 증거로 낼지 선택해야 합니다. 왼쪽이 참이면 `Or.inl`, 오른쪽이 참이면 `Or.inr`를 씁니다. '내가 이쪽 증거를 내겠다'고 선언하는 것입니다."

```lean
-- 왼쪽(2 = 2)이 참 → Or.inl 선택
example : 2 = 2 ∨ 3 = 4 := Or.inl rfl

-- 오른쪽(3 = 3)이 참 → Or.inr 선택
example : 1 = 2 ∨ 3 = 3 := Or.inr rfl
```

> "1교시는 여기서 마무리합니다. 오늘 배운 기호들을 빠르게 복습해 보겠습니다."
>
> "`:` 타입 라벨, `:=` 실제 값, `⊢` 현재 목표, `→` 한 방향 함의, `↔` 양방향 동치, `∧` 둘 다 참, `∨` 하나만 참이어도 됨."
>
> "기호 입력 방법이 궁금한 분은 사전 파일의 '기호 입력 방법 빠른 참조표'를 확인하세요. VS Code에서 `\r`, `\iff`, `\and`, `\or` 이런 식으로 입력하면 됩니다."

## 2교시 — 기호 사전 후반 + 키워드 사전 전반 (60분)

### 0~15분 — A-8 `¬`, A-9 `∀`, A-10 `∃` — 부정과 한정자

#### A-8 `¬` — '아니다', 그런데 사실 `P → False`

> "¬P는 직관적으로 'P가 아니다'입니다. 그런데 Lean에서 ¬P는 `P → False`와 **완전히 동일**합니다."
>
> "이게 처음엔 생뚱맞게 들립니다. 왜 '아니다'가 '→ False'랑 같은 걸까요? 단계별로 풀어보겠습니다."

*(잠깐 멈추며)*

> "먼저 `False`가 뭔지부터 짚겠습니다. Lean에서 `False`는 **절대로 참이 될 수 없는 명제**입니다. '1 = 2이면서 동시에 1 ≠ 2'처럼 영원히 성립 불가능한 것이죠."
>
> "자, 이제 법정 장면을 상상해 봅시다."

*(천천히 시나리오를 펼치며)*

> "검사가 '피의자는 범행 당일 서울에 있었다(P)'고 주장합니다."
>
> "변호사가 이걸 반박하고 싶습니다. 즉 ¬P를 증명하고 싶은 거죠."
>
> "변호사가 말합니다. '좋습니다. 검사 말대로 피의자가 서울에 있었다고 **일단 가정해봅시다**. 그런데 그 시각 피의자가 부산에서 찍힌 CCTV가 있습니다. 한 사람이 동시에 서울과 부산에 있을 수 없습니다. **모순**입니다. 따라서 피의자는 서울에 없었습니다.'"
>
> "이 구조가 정확히 `P → False`입니다."

```
가정: P가 참   (피의자가 서울에 있었다)
      |
      | 논리 전개
      v
False 도출     (동시에 부산에도 있었다 → 불가능 → 모순)
      |
      v
결론: ¬P       (피의자는 서울에 없었다)
```

**Q: '가정했더니 모순이 생긴다'는 것이 왜 '아니다'의 증명이 될까요?**

*(학생들이 생각할 시간 20초)*

> "만약 P가 정말 참이었다면 모순이 생겨선 안 됩니다. 그런데 P가 참이라고 가정했더니 모순이 생겼습니다. 그러면 우리의 가정, 즉 'P가 참이다'가 틀렸다는 것입니다. 따라서 P는 거짓, 즉 ¬P입니다."
>
> "수학에서 이것을 **귀류법(proof by contradiction)**이라고 부릅니다. Lean의 ¬P = P → False는 귀류법을 논리식으로 정확하게 표현한 것입니다."

이제 Lean 코드로 확인해보겠습니다.

```lean
-- ¬(2 = 3) 증명: '2 = 3이 아니다'
example : ¬ (2 = 3) := by
  intro h    -- h : 2 = 3  이라고 '일단 가정한다'
  exact absurd h (by decide)
```

```
[ InfoView — intro h 전후 ]

intro h 전:   ⊢ ¬(2 = 3)
              = ⊢ 2 = 3 → False   ← Lean은 ¬P를 P→False로 읽는다

intro h 후:   h : 2 = 3
              ⊢ False              ← 이제 모순(False)을 만들면 끝!
```

> "보이시나요? `intro h`를 쓰는 순간 목표가 `False`로 바뀝니다. 이제 '2 = 3이라는 가정 h를 가지고 모순을 만들어내라'는 상태가 된 겁니다."
>
> "`decide`는 '2 ≠ 3임을 계산으로 확인'합니다. `absurd`는 'h(2 = 3)와 이 사실(2 ≠ 3)이 동시에 존재하면 모순'임을 선언합니다. 모순이 도출되었으므로 증명 완료입니다."

**한 줄 요약:**

```
¬P = P → False
   = "P가 참이라고 가정하면 반드시 모순이 생긴다"
   = 귀류법의 Lean 버전

증명 절차:
  1. intro h    → P가 참이라고 가정. 목표가 False로 바뀐다.
  2. 모순 도출  → h를 이용해 False를 만들어내면 증명 완료.
```

#### A-9 `∀` — '모든', 예외 없이

> "∀ x, P x는 '모든 x에 대해 P(x)가 성립한다'입니다. '예외 없이 전부 다'라는 주장입니다."
>
> "∀를 증명하는 방법은 놀랍도록 단순합니다. '임의의 하나'에 대해 성립함을 보이면 됩니다. 이것이 수학에서 일반성의 원리입니다."
>
> "Lean에서는 `intro n`을 씁니다. '임의의 x 하나를 골라와라'는 전술입니다."

```lean
theorem zero_add : ∀ x : Nat, x + 0 = x := by
  intro n    -- '임의의 자연수 n 하나를 가져온다'
  rfl        -- n + 0 = n은 정의상 참

-- 이미 증명된 ∀ 정리 사용하기: 특정 값 대입
#check zero_add       -- zero_add : ∀ (x : ℕ), x + 0 = x
#check zero_add 5     -- 5 + 0 = 5 : Prop (5를 대입한 결과)
```

> "모든 백조는 하얗다 (∀ x, 백조(x) → 하얗다(x)). `intro n`은 '임의의 백조 한 마리를 골라오세요'라는 전술입니다. 그 한 마리가 하얀지 보이면, 모든 백조가 하얗다는 게 증명됩니다."
>
> "그런데 여러분, 실제 세계에서 이 논리가 무너진 역사적 사례를 아시나요? 1697년 유럽인들이 호주에서 검은 백조를 발견했습니다. 그때까지 '모든 백조는 하얗다'는 진리처럼 여겨졌죠. 수학에서 ∀는 반례가 하나도 없어야 성립합니다. 엄격합니다."

#### A-10 `∃` — '존재한다', 증거를 내밀어라

> "∃ x, P x는 'P(x)를 만족하는 x가 적어도 하나 존재한다'입니다."
>
> "∃를 증명하는 방법도 단순합니다. 그냥 증거를 하나 내밀면 됩니다. Lean에서는 `use` 키워드를 씁니다."
>
> "3보다 큰 자연수가 존재한다고 주장하려면? 4를 내밀면 끝입니다."

```lean
example : ∃ n : Nat, n > 3 := by
  use 4        -- 4를 증거로 제시한다
  norm_num     -- 4 > 3 자동 증명
```

> "∀는 '예외 없이 모두 그렇다'고 고집하는 말투이고, ∃는 '하나만 있어도 된다'는 설득의 말투입니다."
>
> "∀를 증명하려면 모든 경우를 커버해야 하니 어렵고, ∃를 증명하려면 하나의 증거만 내면 되니 쉽습니다."
>
> "반대로 ∀를 반박하려면 반례 하나만 찾으면 되고, ∃를 반박하려면 그런 것이 하나도 없음을 보여야 합니다."

### 15~28분 — A-11 수 체계 `ℕ ℤ ℝ ℚ` — 숫자에도 등급이 있다

> "Lean에서 숫자는 종류에 따라 다른 기호를 씁니다. 수학 시간에 배우는 수 체계를 그대로 씁니다."

| 기호 | 이름 | 포함하는 수 | Lean 이름 |
|------|------|------------|-----------|
| `ℕ` | 자연수 | 0, 1, 2, 3, ... | `Nat` |
| `ℤ` | 정수 | ...-2,-1,0,1,2,... | `Int` |
| `ℚ` | 유리수 | 분수 포함 | `Rat` |
| `ℝ` | 실수 | √2, π 포함 | `Real` |

> "그런데 여기서 함정이 있습니다. 직접 보여드리겠습니다."

> **[ 반드시 시연 ]**
>
> VS Code에서 다음을 입력한다.
> ```lean

> ```
> **결과: 0** (학생들이 놀란다)
>
> 수학에서 3 - 5 = -2이지만, ℕ에는 음수가 없으므로 0이 된다.  
> 이것을 **'포화 뺄셈(truncated subtraction)'**이라 한다.  
> 이어서 `#eval (3 : Int) - 5`를 입력하면 -2가 나오는 것을 비교해서 보여준다.

> "이 함정 때문에 자연수 뺄셈을 할 때 조심해야 합니다. 코드에서 a - b를 썼을 때 a가 b보다 작으면 0이 나옵니다. 이 사실을 모르면 이상한 결과에 당황하게 됩니다."

#### A-12 `←` — rw에서의 역방향

> "←는 독립적으로 쓰이는 기호가 아닙니다. `rw`(치환) 전술 안에서만 나타납니다."
>
> "`rw [h]`는 h : A = B일 때 목표에서 A를 B로 바꿉니다. 정방향 치환이죠."
>
> "`rw [← h]`는 반대로 목표에서 B를 A로 바꿉니다. 역방향 치환입니다."
>
> "VS Code에서 이 기호를 입력하려면 `\l` 또는 `\leftarrow`라고 입력한 뒤 탭이나 엔터를 누르면 됩니다."

```lean
-- mul_comm a b : a * b = b * a
rw [mul_comm a b]    -- 목표에서 'a*b'를 'b*a'로 바꿈  (정방향)
rw [← mul_comm a b]  -- 목표에서 'b*a'를 'a*b'로 바꿈  (역방향)
```

#### B-3 `theorem` / `lemma` / `example` — 수학 명제 선언

> "이 세 키워드는 모두 수학 명제를 선언하는 데 씁니다. 그런데 중요한 사실이 있습니다."
>
> "Lean 4에서 `theorem`과 `lemma`는 **기능적으로 완전히 동일합니다.** 내부적으로 같은 방식으로 처리됩니다."
>
> "차이는 코드를 읽는 사람에게 주는 신호뿐입니다. `theorem`을 보면 '아, 이게 중요한 최종 결과구나', `lemma`를 보면 '아, 이건 더 큰 증명을 위한 보조 도구구나' 하고 알 수 있습니다."

```lean
-- theorem: 중요한 최종 결과
theorem pythagorean (a b c : ℝ) : a^2 + b^2 = c^2 := sorry

-- lemma: 보조 도구, 중간 단계
lemma mul_zero_eq (a : ℝ) : a * 0 = 0 := by ring

-- example: 이름 없는 연습용
example : 2 + 2 = 4 := rfl
```

| 키워드 | 이름 저장 | 재사용 | 인간에게 보내는 신호 |
|--------|---------|--------|-------------------|
| `theorem` | 저장됨 | 가능 | "중요한 최종 결과" |
| `lemma` | 저장됨 | 가능 | "보조 도구, 중간 단계" |
| `example` | 없음 | 불가능 | "연습용, 데모용" |

> "요리 레시피로 비유하면, `theorem`은 완성된 메인 요리이고 `lemma`는 그 요리를 만들기 위한 소스 만들기, 재료 손질 같은 단계입니다. `example`은 손님에게 내기 전에 맛을 보는 시식 연습입니다."

### 52~60분 — B-4 `by` — 단계별 증명의 시작

> "`by`는 '지금부터 전술(tactic)을 써서 단계적으로 증명하겠다'는 선언입니다."
>
> "Lean에서 증명을 쓰는 방법은 크게 두 가지입니다. `by` 없이 `:=` 뒤에 한 번에 증명을 쓰는 **항 모드(term mode)**와, `by` 이후에 전술들을 한 줄씩 써나가는 **전술 모드(tactic mode)**입니다."

```lean
-- 항 모드: 증명을 직접 하나의 값으로
theorem t1 : 2 + 2 = 4 := rfl

-- 전술 모드: by 이후 전술들을 단계별로
theorem t2 : 2 + 2 = 4 := by
  rfl

-- 복잡한 증명은 전술 모드가 훨씬 편하다
theorem t3 (a b : ℝ) (h : a ≤ b) : a ≤ b + 1 := by
  linarith
```

> "증명이 간단하면 항 모드로, 여러 단계가 필요하면 전술 모드로 씁니다. 처음에는 전술 모드가 훨씬 읽기 쉽고 디버깅도 쉽습니다. `by`를 쓰는 순간 오른쪽 InfoView에 ⊢(목표)가 나타납니다."
>
> "비유하자면, 항 모드는 '완성된 요리를 한 번에 제출', 전술 모드는 '1단계 재료 준비, 2단계 볶기, 3단계 플레이팅'처럼 단계별로 만드는 것입니다. 처음 배울 때는 전술 모드로 쓰는 것이 훨씬 유익합니다."

### 0~12분 — B-5 `rfl` — '양쪽이 계산하면 같다'

> "`rfl`은 reflexivity(반사성)의 줄임말입니다. '양쪽을 계산하면 정확히 같은 값이 된다'는 것을 증명하는 가장 단순한 전술입니다."
>
> "작동 방식을 한 문장으로 설명하면, Lean이 등호 양쪽을 직접 계산하고 결과가 같으면 '응, 같네'라고 증명을 끝냅니다."

```lean
-- rfl이 되는 경우: 계산하면 양쪽이 같아짐
theorem ex1 : 2 + 2 = 4 := rfl          -- 2+2 계산 → 4, 같음
theorem ex2 : [1,2] ++ [3] = [1,2,3] := rfl  -- 리스트 연결도 됨

-- rfl이 안 되는 경우: 변수가 있어서 계산 불가
-- theorem ex3 (a b : Nat) : a + b = b + a := rfl  -- 오류!
-- a, b가 구체적인 숫자가 아니니 계산이 안 된다.
-- 이 경우 ring이나 omega를 써야 한다.
```

```
[ rfl 전후 InfoView 변화 ]

rfl 전:   ⊢ 2 + 2 = 4
rfl 후:   goals accomplished  (증명 완료!)
```

> "저울 비유를 써보겠습니다. 등호 왼쪽 접시에 2+2를 올리고, 오른쪽 접시에 4를 올렸습니다. Lean이 양쪽을 계산해서 무게가 같으면 '균형 잡혔다'고 판정하는 것입니다. 2+2는 4가 되고, 4는 4이므로 균형."
>
> "변수가 있으면 계산을 못 합니다. a+b를 계산하려면 a와 b가 구체적인 숫자여야 하는데, 변수는 '어떤 수든 될 수 있으므로' 결정이 안 됩니다. 그래서 rfl 오류가 납니다."

### 12~28분 — B-6 `rw` — 스티커 바꿔치기 (슈퍼포지션)

#### 원리: 등호를 이용한 치환

> "`rw`는 rewrite(다시 쓰기)의 줄임말입니다. 수학에서는 치환(대입), 멋진 말로는 슈퍼포지션(superposition)이라고 합니다."
>
> "원리는 간단합니다. `h : A = B`라는 사실을 알고 있을 때, 목표에 있는 A를 B로 바꿉니다. 종이에 적힌 A라는 글자 위에 B라는 스티커를 붙이는 겁니다. 스티커가 붙으면 원래 글자 대신 스티커 내용이 보입니다."

```lean
-- rw 사용 형태
rw [h]           -- h : A = B  →  목표에서 A를 B로 바꿈
rw [← h]         -- h : A = B  →  목표에서 B를 A로 바꿈 (역방향)
rw [h] at hyp    -- 목표가 아닌 가정 hyp에서 치환
rw [h1, h2, h3]  -- 여러 치환을 한 줄에 연속으로
```

#### 단계별 시연

> "구체적인 예를 단계별로 따라가 보겠습니다. 목표는 `a * b * c = b * (a * c)`를 증명하는 것입니다."

```lean
example (a b c : ℝ) : a * b * c = b * (a * c) := by
  rw [mul_comm a b]    -- 1단계: a*b → b*a
  rw [mul_assoc b a c] -- 2단계: b*a*c → b*(a*c)
```

```
[ InfoView 커서별 상태 변화 ]

by 뒤 클릭:                ⊢ a * b * c = b * (a * c)
rw [mul_comm a b] 뒤:     ⊢ b * a * c = b * (a * c)
                              ↑ a와 b의 위치가 바뀌었다!
rw [mul_assoc b a c] 뒤:  ⊢ b * (a * c) = b * (a * c)
                           -- 양쪽이 같으므로 자동 완료
최종:                      No goals
```

> "커서를 직접 옮겨가며 InfoView가 어떻게 바뀌는지 실시간으로 확인해 보겠습니다."

> **[ 반드시 시연 ]**
>
> 위 코드를 VS Code에 입력하고 커서를 `by` 뒤 / 첫 번째 `rw` 뒤 / 두 번째 `rw` 뒤로 순서대로 이동하면서 InfoView 변화를 화면 공유로 보여준다.  
> 학생들도 직접 따라 입력하게 한다. (5분 소요)

#### rw의 한계와 ring의 등장

> "그런데 여러분, 생각해 보세요. 모든 대수 등식을 이렇게 `rw`로 하나하나 치환하면 너무 피곤하지 않을까요? 위 예시는 두 단계밖에 없었지만 복잡한 식이라면 열 단계, 스무 단계도 될 수 있습니다."
>
> "그래서 Lean 4에는 `ring`이라는 자동 전술이 있습니다. 대수적 등식이라면 한 줄로 끝냅니다."

```lean
import Mathlib.Tactic

-- rw로 하면 여러 줄인 것을...
example (a b c : ℝ) : a * b * c = b * (a * c) := by
  ring   -- 한 줄로 끝!

-- 더 복잡한 식도 ring 한 줄로
example (a b : ℝ) : (a + b)^2 = a^2 + 2*a*b + b^2 := by
  ring
```

> "ring은 환(Ring) 구조에서의 모든 등식, 즉 교환법칙, 결합법칙, 분배법칙이 얽힌 어떤 복잡한 등식도 한 방에 해결합니다."

> **[ 주의 ]**
>
> `ring`은 **등식(=)만** 해결한다. 부등식(`≤`, `<`)이 나오면 ring을 쓸 수 없다. 그때는 `linarith`를 써야 한다.  
> 그리고 `rw`에 넣는 h는 반드시 `A = B` 또는 `A ↔ B` 형태여야 한다. `A → B`는 `rw`에 쓸 수 없다.

### 28~40분 — B-7 `exact`, B-8 `intro`, B-9 `apply`

#### B-7 `exact` — '바로 이것이 답'

> "`exact h`는 '현재 증명해야 할 목표와 h가 타입이 정확히 일치한다. h가 바로 증명이다'라는 전술입니다."
>
> "퍼즐 조각을 맞추는 것과 같습니다. 목표라는 구멍의 모양과 내가 가진 조각 h의 모양이 정확히 같을 때만 쓸 수 있습니다. 모양이 비슷하지만 조금이라도 다르면 Lean이 `type mismatch` 오류를 냅니다."

```lean
example (a b : ℝ) (h : a ≤ b) : a ≤ b := by
  exact h    -- 목표(a ≤ b)와 h의 타입(a ≤ b)이 정확히 같음 → 끝!

-- 조금이라도 다르면 오류
-- example (a b c : ℝ) (h : a ≤ b) : a ≤ c := by
--   exact h    -- 오류! a ≤ b 와 a ≤ c 는 다르다
```

---

#### B-8 `intro` — '가정하자'

> "목표가 `A → B` 또는 `∀ x, P x` 형태일 때 `intro`를 씁니다."
>
> "목표가 `A → B`일 때 `intro h`를 실행하면 두 가지 일이 동시에 일어납니다. 첫째, A가 참이라는 가정이 h라는 이름으로 가정 목록(⊢ 위)에 추가됩니다. 둘째, 목표가 B로 바뀝니다."
>
> "'비가 온다고 치자. 그러면 땅이 젖는다.' `intro h`는 '비가 온다고 치자' 부분을 처리하고, 이후는 '땅이 젖는다'만 증명하면 되는 상태로 만듭니다."

```lean
example (a b : ℝ) : a ≤ b → a + 1 ≤ b + 1 := by
  -- by 뒤 상태:   ⊢ a ≤ b → a + 1 ≤ b + 1
  intro h
  -- intro h 뒤:   h : a ≤ b
  --               ⊢ a + 1 ≤ b + 1
  linarith  -- h : a ≤ b를 이용해 자동 증명

-- ∀ 에서도 intro 사용
example : ∀ n : Nat, n + 0 = n := by
  intro n     -- 임의의 n 하나를 가져옴
  rfl         -- n + 0 = n은 정의상 참
```

```
[ intro h 전후 비교 ]

intro h 전:   ⊢ a ≤ b → a + 1 ≤ b + 1

intro h 후:   h : a ≤ b
              ⊢ a + 1 ≤ b + 1
              (화살표가 사라지고 가정이 추가되었다!)
```

#### B-9 `apply` — '이 정리를 역방향으로 적용'

> "`apply h`는 정리 h의 결론이 현재 목표와 일치할 때, 목표를 h의 전제 조건으로 바꾸는 전술입니다."
>
> "역방향이라고 표현하는 이유는, 정리를 앞에서 뒤로 쓰는 것이 아니라 뒤에서 앞으로 거슬러 올라가기 때문입니다. '이 결론을 얻으려면 무엇이 필요한가?'를 물어서 전제를 새로운 목표로 만드는 것입니다."

```lean
-- add_le_add : a ≤ b → c ≤ d → a + c ≤ b + d
-- '이 정리를 쓰려면 a ≤ b와 c ≤ d 두 가지가 필요하다'

example (a b c d : ℝ) (h1 : a ≤ b) (h2 : c ≤ d) : a + c ≤ b + d := by
  apply add_le_add  -- 목표가 두 개의 전제로 쪼개짐
  · exact h1        -- 목표 1: ⊢ a ≤ b
  · exact h2        -- 목표 2: ⊢ c ≤ d
```

```
[ apply add_le_add 전후 ]

apply 전:   ⊢ a + c ≤ b + d

apply 후:   2 goals
            ⊢ a ≤ b
            ⊢ c ≤ d
```

### 40~52분 — B-10 `constructor`, B-11 `cases`, B-12 자동 전술

#### B-10 `constructor` — '둘로 쪼갠다'

> "목표가 `A ∧ B` 또는 `A ↔ B` 형태일 때 `constructor`를 씁니다. 두 개의 독립적인 목표로 쪼개어 각각 증명합니다."
>
> "`A ∧ B`를 증명하려면 A도 참이고 B도 참임을 각각 보여야 합니다. `constructor`는 그 분리를 Lean에게 알리는 선언입니다."

```lean
-- ∧ 증명
example : 2 > 1 ∧ 3 > 2 := by
  constructor      -- 두 목표로 쪼갠다
  · norm_num       -- 첫 번째: ⊢ 2 > 1
  · norm_num       -- 두 번째: ⊢ 3 > 2

-- ↔ 증명 (두 방향 각각)
example (a b : ℝ) : a + b = b + a ↔ True := by
  constructor
  · intro _; trivial   -- 정방향: a + b = b + a → True
  · intro _; ring      -- 역방향: True → a + b = b + a
```

---

#### B-11 `cases` — '경우를 나눈다'

> "`cases h`는 h의 형태에 따라 경우를 나눠서 각 경우를 따로 증명하는 전술입니다. `h : P ∨ Q`가 있을 때, P가 참인 경우와 Q가 참인 경우로 나눠서 각각 결론을 증명합니다."
>
> "수학에서 '경우를 나누어 증명한다'는 방식이 있죠. Lean에서 그 방식이 바로 `cases`입니다."

```lean
-- h : P ∨ Q 일 때, P든 Q든 R이 성립함을 증명
example (P Q R : Prop) (h : P ∨ Q)
    (hp : P → R) (hq : Q → R) : R := by
  cases h with
  | inl hp' => exact hp hp'   -- P인 경우: hp'가 P의 증거
  | inr hq' => exact hq hq'   -- Q인 경우: hq'가 Q의 증거
```

---

#### B-12 자동 증명 전술 — ring / linarith / norm_num / omega / decide / simp

> "이제 여러분의 삶을 편하게 해줄 자동 전술들을 정리합니다. 언제 어느 것을 쓰느냐가 핵심입니다."

```lean
-- ring: 대수 등식 (교환·결합·분배법칙 조합)
example (a b : ℝ) : (a + b)^2 = a^2 + 2*a*b + b^2 := by ring

-- linarith: 선형 부등식 (가정에서 부등식 결론 도출)
example (a b c : ℝ) (h1 : a ≤ b) (h2 : b ≤ c) : a ≤ c := by linarith

-- norm_num: 구체적 수치 계산 검증
example : (2 : ℝ)^10 = 1024 := by norm_num

-- omega: 자연수/정수 선형 산술 (나머지 연산 포함)
example (n : Nat) (h : n % 2 = 0) : (n + 2) % 2 = 0 := by omega

-- decide: 유한하게 계산 가능한 명제
example : 2 < 5 := by decide

-- simp: 기본 규칙 모음으로 자동 단순화
example (a : ℝ) : a + 0 = a := by simp
```

| 전술 | 해결하는 것 | 예시 |
|------|-----------|------|
| `ring` | 환(ring)의 등식 | `(a+b)^2 = a^2+2ab+b^2` |
| `linarith` | 선형 부등식 | `a ≤ b, b ≤ c → a ≤ c` |
| `norm_num` | 구체적 수치 계산 | `2^10 = 1024` |
| `omega` | 자연수/정수 선형 산술 | `n % 2 = 0 → (n+2) % 2 = 0` |
| `decide` | 유한하게 계산 가능한 명제 | `2 < 5` |
| `simp` | 자동 단순화 | `a + 0 = a` |

> "선택 기준을 한 줄로 정리하면: 등식이면 `ring`, 부등식이면 `linarith`, 구체적 숫자면 `norm_num`, 정수 나머지면 `omega`, 단순 참거짓이면 `decide`."

> **[ ring 쾌감 시연 ]**
>
> 먼저 `(a+b)^2 = a^2 + 2ab + b^2`를 `rw`로 하나하나 전개하는 시도를 보여준다.  
> 그 다음 같은 것을 `ring` 한 줄로 끝내는 것을 보여준다.  
> 학생들의 탄식과 감탄을 유도한다.

### 52~60분 — B-13 `sorry`, B-14 `calc` — 미완성 표시와 단계별 계산

#### B-13 `sorry` — 증명의 빈자리

> "`sorry`는 증명을 아직 완성하지 못했을 때 임시로 쓰는 자리 표시자입니다. 마치 시험지에 '풀이 생략, 나중에 채움'이라고 적어두는 것과 같습니다."
>
> "Lean이 `sorry`를 발견하면 이 부분은 일단 통과시켜주는 대신, 노란 경고를 표시합니다. `declaration uses sorry`라는 메시지가 그것입니다."
>
> "연습 문제를 할 때 `sorry`를 써놓고 시작하는 것이 좋은 방법입니다. 코드 구조를 먼저 잡아두고, `sorry`를 하나씩 실제 증명으로 채워나가는 방식으로 작업합니다."

```lean
-- sorry를 쓰면 Lean이 일단 통과시켜 준다
theorem difficult_theorem (a b : ℝ) : a^2 + b^2 ≥ 0 := by
  sorry   -- 노란 경고: 'declaration uses sorry'

-- 연습 문제 구조: sorry를 먼저 채우고 나중에 실제 증명으로 교체
example (a b : ℝ) (h : a ≤ b) : Real.exp a ≤ Real.exp b := by
  sorry   -- 여기를 채워야 함. 힌트: Real.exp_le_exp.mpr h
```

> **[ 주의 ]**
>
> `sorry`가 포함된 증명은 신뢰할 수 없다.  
> `sorry`는 Lean이 '이 부분은 참이다'고 가정하도록 강제하는 것이다.  
> 거짓 명제도 `sorry`를 쓰면 증명된 것처럼 보인다.  
> 최종 제출에서는 `sorry`가 하나도 없어야 한다.  
> 코드베이스 전체에서 `sorry`를 검색해서 없앴는지 반드시 확인한다.

#### B-14 `calc` — 풀이 과정을 단계별로

> "`calc`는 calculation의 줄임말입니다. 수학 시험에서 풀이 과정을 한 줄씩 적는 것처럼, 등식이나 부등식의 변환 과정을 단계별로 나열하는 블록입니다."
>
> "여러 등식을 연결할 때 `calc`를 쓰면 각 단계가 무엇을 하는지 명확하게 드러납니다. `_` 기호는 '이전 줄의 결과'를 가리킵니다."

```lean
example (a b : ℝ) : (a + b)^2 = a^2 + 2*a*b + b^2 := by
  calc (a + b)^2
      = (a + b) * (a + b)       := by ring
    _ = a*a + a*b + b*a + b*b   := by ring
    _ = a^2 + 2*a*b + b^2       := by ring
-- _ 는 '이전 줄의 결과'를 가리킨다.
```

> "실제로는 `ring` 한 줄로 끝나지만, 등식과 부등식이 섞인 긴 추론을 할 때 `calc`로 쓰면 논리의 흐름이 그대로 드러납니다."

### 파트 C — InfoView 화면 완전 해독

> "지금까지 전술을 배우면서 InfoView를 많이 언급했습니다. 이제 InfoView 화면 구조를 체계적으로 정리해 드리겠습니다."
>
> "InfoView는 Lean 4에서 가장 중요한 피드백 창입니다. 현재 증명의 상태를 실시간으로 보여줍니다."

```
[ InfoView 화면 완전 구조 ]

2 goals                    ← 아직 증명할 목표가 2개 남았다

case left                  ← 목표의 이름 (constructor로 쪼갤 때 생김)
a b : ℝ                    ← 가정: a, b는 실수이다
h : a ≤ b                  ← 가정: a ≤ b라는 증거 h가 있다
⊢ a + 1 ≤ b + 1            ← 목표: 이것을 증명해야 한다

case right                 ← 두 번째 목표
a b : ℝ
⊢ b - 1 ≤ a → False
```

> "화면을 요소별로 읽는 법입니다."
>
> "맨 위 'N goals'는 현재 해결해야 할 목표가 N개 남았다는 뜻입니다. 0개가 되거나 `goals accomplished`가 뜨면 증명 완료입니다."
>
> "'case 이름'은 `constructor`나 `cases`로 목표가 쪼개질 때 각 목표에 붙는 이름입니다. 어느 목표에 집중하고 있는지 알 수 있습니다."
>
> "⊢ 위에 있는 모든 것들, 이름 : 타입 형태의 줄들은 가정입니다. 이미 알고 있는 사실들이자 증명에 사용할 수 있는 재료입니다."
>
> "⊢ 아래에 있는 것이 현재 증명해야 할 목표입니다. 전술을 쓸 때마다 이것이 바뀌거나 사라집니다."

> **[ InfoView 활용법 심화 ]**
>
> 오류가 발생했을 때 InfoView를 확인하는 것이 가장 먼저 해야 할 일이다.  
> 목표가 예상과 다른 형태라면, 앞 전술에서 예상치 못한 변환이 일어난 것이다.  
> 특히 `simp`를 쓴 뒤에는 InfoView를 반드시 확인해야 한다.  
> 여러 목표가 있을 때 `·` 없이 전술을 쓰면 Lean이 혼란스러워하므로 InfoView의 'case' 이름을 보고 현재 어느 목표를 보고 있는지 확인한다.

### 파트 D — 오류 메시지 해독: '빨간 줄은 적이 아니다'

> "Lean 4에서 오류 메시지가 뜨면 많은 학생들이 당황합니다. 하지만 오류 메시지는 사실 가장 친절한 선생님입니다. '여기가 문제야, 이걸 고쳐야 해'라고 정확히 짚어주거든요."

---

#### `unknown identifier 'X'`

> "X라는 이름을 찾을 수 없다는 뜻입니다. 원인은 대부분 세 가지 중 하나입니다."
>
> "첫째, 오타. `mul_comm`을 `mul_Comm`처럼 대소문자가 틀린 경우입니다."
>
> "둘째, 해당 정리가 있는 라이브러리를 `import`하지 않은 경우입니다."
>
> "셋째, 정리 이름 자체를 잘못 알고 있는 경우입니다."
>
> "해결 방법: `#check 이름`을 입력해서 정확한 이름을 찾아보거나, 라이브러리 문서를 확인합니다."

```lean
-- 오류 예시
-- rw [mul_Comm a b]   -- 'C'가 대문자 → 오류

-- 올바른 방법: 먼저 확인
#check mul_comm   -- mul_comm : ∀ (a b : ?α), a * b = b * a
-- 이름이 mul_comm (소문자 c) 임을 확인 후 사용
```

#### `type mismatch`

> "타입이 안 맞는다는 뜻입니다. Lean이 기대하는 타입과 실제로 제공한 타입이 다를 때 납니다."
>
> "가장 흔한 경우는 `exact`를 쓸 때 목표의 타입과 제공한 증거의 타입이 다른 경우, 그리고 변수의 타입을 잘못 선언한 경우입니다."
>
> "해결 방법: InfoView에서 목표의 정확한 타입을 확인하고, 제공하는 증거의 타입을 `#check`로 확인합니다."

> **[ 오류 유도 시연 ]**
>
> 다음 코드를 입력해서 `type mismatch` 오류를 의도적으로 만든다.
> ```lean
> example (a b c : ℝ) (h : a ≤ b) : a ≤ c := by
>   exact h   -- a ≤ b와 a ≤ c는 다르다
> ```
> 오류 메시지를 읽으면서 'expected type'과 'given type'을 비교한다.  
> 학생들이 오류 메시지를 읽는 연습이 이 시연의 핵심이다.

#### `unsolved goals`

> "증명이 끝나지 않았다는 뜻입니다. 전술을 다 썼는데도 아직 해결되지 않은 목표가 남아있을 때 납니다."
>
> "원인은 전술이 부족하거나, 전술 순서가 잘못된 경우가 대부분입니다."
>
> "해결 방법: InfoView를 확인해서 어떤 목표가 남아있는지 파악하고, 적절한 전술을 추가합니다. 일단 `sorry`로 임시 처리하고 나중에 채울 수도 있습니다."

---

#### `declaration uses 'sorry'`

> "`sorry`가 포함되어 있다는 경고입니다. 오류가 아니라 경고이므로 코드는 돌아가지만, 신뢰할 수 없다는 표시입니다."
>
> "`sorry`를 쓴 채로 진행하는 것은 개발 과정에서 일반적이지만, 최종적으로는 반드시 실제 증명으로 교체해야 합니다."
>
> "과제나 프로젝트 제출 전에 파일 전체에서 `sorry`를 검색해서 하나도 없는지 확인하는 것이 좋습니다."

## 마무리 — 전체 요약 및 사전 활용 안내

> "오늘 3교시에 걸쳐 `lean4_tutorial_dict.md`의 모든 항목을 함께 읽었습니다. 처음에는 암호처럼 보이던 기호들이 이제는 친숙하게 느껴지길 바랍니다."
>
> "중요한 것은 이 모든 것을 지금 다 외우는 것이 아닙니다. 이 사전 파일이 어디에 있는지 알고, 모르는 것이 나왔을 때 바로 찾아볼 수 있으면 됩니다."
>
> "마지막으로 핵심 기호와 키워드를 빠르게 복습하겠습니다."

```
기호 편:
  :     콜론        = 타입 라벨 (왼쪽은 오른쪽 타입에 속한다)
  :=    콜론-등호   = 실제 값/증명 대입
  ⊢     턴스타일    = 현재 증명 목표 (⊢ 아래가 목표)
  →     화살표      = 이면 / 함수 타입 (한 방향!)
  ↔     양방향      = 동치 (if and only if, .mp .mpr으로 분해)
  ∧     논리 곱     = 그리고 (and, 둘 다 참)
  ∨     논리 합     = 또는 (or, 하나만 참이어도)
  ¬     부정        = 아니다 (= P → False)
  ∀     전칭        = 모든 ~에 대해 (intro로 증명)
  ∃     존재        = ~가 존재한다 (use로 증명)
  ℕℤℝℚ  수 체계     = ℕ 주의: 음수 없음!
  ←     역방향      = rw [← h]로 사용
  ·     가운뎃점    = 목표 분리 증명 시 사용
  _     언더스코어  = calc 이전 결과 / 인수 무시

키워드 편:
  #check / #eval     탐문 도구 (증명에 포함 안 됨)
  def                이름 붙여 정의
  theorem / lemma    명제 선언 (기능 동일, 신호만 다름)
  example            이름 없는 연습용
  by                 전술 모드 시작
  rfl                계산하면 양쪽이 같다
  rw                 치환 (스티커 바꿔치기)
  exact              이것이 바로 답
  intro              가정 도입 (→ 의 왼쪽을 가정으로)
  apply              정리를 역방향 적용
  constructor        ∧ 또는 ↔ 를 둘로 쪼갠다
  cases              ∨ 등 경우를 나눠서 증명
  sorry              미완성 자리 표시 (나중에 채울 것!)
  calc               단계별 변환 과정 나열

자동 전술:
  ring        대수 등식 (교환·결합·분배)
  linarith    선형 부등식
  norm_num    구체적 수치 계산
  omega       정수/자연수 선형 산술
  decide      유한 계산 가능한 명제
  simp        기본 규칙으로 자동 단순화
```

### 시연 필수 목록

1. **InfoView 커서 이동**: `by` 뒤 → 각 전술 줄 끝 → 마지막 줄. 목표가 단계별로 어떻게 변하는지 실시간으로 보여준다.
