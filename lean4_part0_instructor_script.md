## 1.3 Lean 4로 "4는 짝수다" 증명하기 (25분)

*(VS Code에 빈 파일을 열어둔다.)*

"드디어 첫 번째 Lean 4 코드입니다. 겁내지 마세요. 지금부터 한 단어씩 천천히 뜯어봅니다."

---

### 1단계: 정의 — `def Even`

*(코드를 천천히 입력하며)*

```lean
def Even (n : Nat) : Prop := ∃ k, n = 2 * k
```

"이 한 줄이 '짝수'라는 개념 전체를 정의합니다. 놀랍지 않나요? 수학 교과서에서 반 페이지에 걸쳐 설명하는 내용이 한 줄입니다."

"한 단어씩 읽어보겠습니다."

```
def   Even   (n : Nat)   : Prop   :=   ∃ k, n = 2 * k
 ↑      ↑        ↑           ↑      ↑          ↑
 │      │        │           │      │          └── "n = 2×k인 k가 존재한다"
 │      │        │           │      └── "실제 내용은"
 │      │        │           └── 이건 명제(참/거짓 주장)
 │      │        └── n은 자연수(Nat)
 │      └── 이 함수의 이름
 └── 새로운 것을 정의하겠다
```

*(각 부분을 짚으며 천천히)*

"`def` — '새로운 것을 정의하겠다'는 선언입니다."

"`Even` — 이름표입니다. 나중에 이 이름으로 불러 씁니다."

"`(n : Nat)` — '입력 재료는 자연수 n'입니다. 콜론(`:`)은 '이것의 종류는'이라는 뜻입니다. n의 종류가 Nat, 즉 자연수라는 것입니다."

"`: Prop` — 이 함수가 돌려주는 결과물의 종류는 명제(Prop)입니다. 참/거짓을 따질 수 있는 문장이라는 뜻입니다."

"`:=` — '실제 내용은'입니다. 주민등록증에서 '직업: 개발자'처럼, 콜론이 종류를 나타내고 콜론-등호가 실제 내용을 나타냅니다."

"`∃ k, n = 2 * k` — '2를 곱해서 n이 되는 k가 존재한다.' ∃는 'exists(존재한다)'의 수학 기호입니다."

*(잠깐 전체를 바라보며)*

"합쳐서 읽으면? **'n이 짝수라는 것은, n = 2×k인 자연수 k가 어딘가에 존재한다는 뜻이다.'** 수학 교과서의 짝수 정의와 완벽하게 일치합니다."

---

**Q: ∃ 기호를 처음 보는 분 계세요?**

"기억 안 나도 전혀 괜찮습니다. 이런 기호들은 `lean4_tutorial_dict.md` 사전을 열면 바로 찾을 수 있습니다. A-10번 항목에 있습니다. 오늘 수업 목표는 기호를 외우는 게 아닙니다. 코드의 흐름을 읽는 것입니다."

### 2단계 + 3단계: 주장 + 증명 — `theorem four_is_even`

*(코드를 입력하며)*

```lean
theorem four_is_even : Even 4 := by
  use 2
  rfl
```

"이 세 줄이 '4는 짝수다'의 완전한 증명입니다."

*(웃으며)*

"믿기 어려울 만큼 짧죠? 하지만 이 안에 수학적 논리가 완전하게 담겨 있습니다. 한 줄씩 뜯어보겠습니다."

---

#### 첫 줄: `theorem four_is_even : Even 4 := by`

```
theorem   four_is_even   : Even 4   := by
   ↑             ↑            ↑        ↑
   │             │            │        └── 전술(tactic)로 증명하겠다
   │             │            └── 증명할 명제: "4는 짝수다"
   │             └── 이 정리의 이름
   └── 정리를 선언한다
```

"`theorem` — '정리를 선언하겠다'입니다. 판사에게 '저 오늘 이 사건 변호할게요'라고 선언하는 것과 같습니다."

"`four_is_even` — 이 정리의 이름입니다. 나중에 다른 증명에서 이 이름을 꺼내 쓸 수 있습니다."

"`: Even 4` — 증명할 명제입니다. 'Even(4)', 즉 '4는 짝수다'를 증명하겠다는 것입니다."

"`:= by` — 전술(tactic)을 써서 단계적으로 증명하겠다는 선언입니다. `by` 뒤에 증명 단계들이 옵니다."

---

> **TIP — 시연 포인트**
>
> `by` 키워드 뒤에 커서를 놓는다.
> 오른쪽 InfoView 창에 `⊢ Even 4`, 즉 '4가 짝수임을 증명해야 한다'는 목표가 나타난다.
> 학생들에게 "이 ⊢ 기호가 지금 우리가 증명해야 할 것을 가리킨다"고 설명한다.
> ⊢ 기호에 대한 자세한 설명은 `lean4_tutorial_dict.md` A-3 항목에 있다.

---

#### 둘째 줄: `use 2`

"자, 실제 증명 시작입니다."

"Even 4의 정의를 다시 떠올려봐요."

```
Even 4 = ∃ k, 4 = 2 * k
       = "4 = 2 × k인 k가 존재한다"
```

"'존재한다(∃)'를 증명하려면 어떻게 해야 할까요?"

*(학생들에게 묻는다.)*

**Q: 'k가 존재한다'는 것을 어떻게 증명하죠?**

*(학생들이 생각할 시간 15초)*

"그렇습니다! 실제로 하나를 보여주면 됩니다. '여기 있어요, 이게 그 k예요.'라고 내미는 것이죠."

"`use 2`는 바로 그겁니다. 'k = 2를 쓰겠습니다'라고 선언하는 겁니다."

"이 순간 Lean의 목표가 바뀝니다. 직접 InfoView에서 확인해 보겠습니다."

*(커서를 `use 2` 앞뒤로 이동하며)*

```
use 2 실행 전 목표:   ⊢ ∃ k, 4 = 2 * k
use 2 실행 후 목표:   ⊢ 4 = 2 * 2
```

"보이시나요? '어딘가에 k가 존재한다'는 막막한 목표가, 'k = 2일 때 4 = 2 * 2가 성립한다'는 구체적인 목표로 바뀌었습니다. 이제 이것만 증명하면 끝이에요."

---

> **TIP — 시연 포인트**
>
> VS Code에서 `use 2` 줄에 커서를 두기 전과 후의 InfoView를 보여준다.
> - `by` 바로 뒤 커서: `⊢ Even 4` 또는 `⊢ ∃ k, 4 = 2 * k`
> - `use 2` 줄 끝 커서: `⊢ 4 = 2 * 2`
> 이 변화를 천천히 보여주는 것이 이 수업의 핵심 시연이다.

---

#### 셋째 줄: `rfl`

"목표가 `4 = 2 * 2`입니다. 어떻게 증명할까요?"

*(학생들이 생각할 시간 5초)*

"Lean에게 시키면 됩니다. 2 * 2를 직접 계산해봐요. 4죠? 그러면 `4 = 4`입니다. 같은 건 같습니다. 증명 끝."

"이 '같은 건 같다'를 처리하는 전술이 `rfl`입니다. **Reflexivity(반사성)**의 약자예요."

*(유머 섞어)*

"'2 * 2가 4인 걸 증명하라'고 하면 저도 좀 황당하긴 합니다. 그런데 Lean은 모든 단계를 명시적으로 요구하거든요. `rfl` 덕분에 '계산해보니 같더라'는 황당한 증명을 한 단어로 끝낼 수 있습니다."

---

> **TIP — 시연 포인트**
>
> `rfl` 줄 끝에 커서를 놓으면 InfoView에 `No goals`가 표시된다.
> 이 순간을 학생들이 직접 볼 수 있게 한다.
> "빨간 줄이 없고 No goals가 뜨면 증명 완료입니다. 박수 치세요."

---

### 전체 흐름 한눈에 보기

*(화면에 흐름도를 보여주며)*

```
목표: Even 4
```

## 1.4 `rfl` 완전 정복 (12분)

"방금 `rfl`을 썼는데, 이게 언제 되고 언제 안 되는지 제대로 알아두어야 합니다. 안 그러면 나중에 안 되는 상황에서 계속 `rfl`만 시도하다가 좌절하게 됩니다."

*(웃으며)*

"제 학생들이 가장 많이 겪는 패턴이 이겁니다. '왜 rfl이 안 되지?' 하면서 화면을 30분 바라보는 것요."

---

### `rfl`이 통하는 경우

"원리는 하나입니다. **Lean이 양쪽을 직접 계산해서 같은 값이 나오면 증명 완료.**"

*(예시들을 하나씩 입력하며)*

```lean
-- 덧셈
theorem ex1 : 2 + 3 = 5 := rfl
```

```
Lean의 내부 독백:
  왼쪽 '2 + 3'을 계산하면... 5
  오른쪽은... 5
  5 = 5. 같은 건 같다. rfl 통과!
```

```lean
-- 곱셈
theorem ex2 : 3 * 4 = 12 := rfl
```

```lean
-- 리스트 길이
theorem ex3 : [1, 2, 3].length = 3 := rfl
```

```lean
-- 함수 적용
def double (n : Nat) : Nat := 2 * n

theorem ex4 : double 5 = 10 := rfl
```

```
Lean의 내부 독백:
  왼쪽 'double 5'를 계산하면...
  double의 정의가 2 * n이니까 2 * 5 = 10
  오른쪽은... 10
  10 = 10. 통과!
```

---

### `rfl`이 안 통하는 경우 — 함정 주의

*(일부러 오류를 만들며)*

```lean
-- 이건 오류가 난다!
-- theorem ex5 (a b : Nat) : a + b = b + a := rfl
```

"왜 안 될까요?"

**Q: rfl이 실패하는 이유를 생각해 보세요.**

*(학생들이 생각할 시간 15초)*

```
Lean의 내부 독백:
  왼쪽 'a + b'를 계산하려는데...
  a가 3인지, 100인지, 0인지 모른다.
  계산 불가.
  rfl 실패!
```

"변수가 있으면 계산을 못 합니다. Lean은 구체적인 숫자가 있어야 계산할 수 있거든요."

"이런 경우에는 다른 전술을 써야 합니다."

```lean
-- 해결책 1: 이미 증명된 정리를 쓴다
theorem ex5 (a b : Nat) : a + b = b + a := Nat.add_comm a b

-- 해결책 2: ring 전술을 쓴다
theorem ex5' (a b : Nat) : a + b = b + a := by ring
```

"`ring`은 대수적 등식이라면 변수가 있어도 자동으로 증명하는 강력한 전술입니다. 나중에 자세히 다룹니다."

---

**정리: `rfl` 사용 가능 조건**

| 상황 | `rfl` 가능? | 대안 |
|------|------------|------|
| 양쪽이 구체적 숫자 | 가능 | — |
| 양쪽이 계산 가능한 식 | 가능 | — |
| 변수가 포함된 일반 식 | 불가 | `ring`, `omega` 등 |
| 가정이 필요한 부등식 | 불가 | `linarith` 등 |

## 1.5 `def`와 `theorem`의 차이 (10분)

"이제 두 핵심 키워드의 차이를 명확히 짚겠습니다. 코드를 처음 볼 때 가장 많이 헷갈리는 부분입니다."

---

### `def` — 값이나 함수를 정의

```lean
def myNumber : Nat := 42
```

"`def`는 '이런 것이 있다'를 선언하는 겁니다. `myNumber`에 42라는 값을 붙여두는 것이죠."

```lean
def double (n : Nat) : Nat := 2 * n

#eval double 5    -- 10
#eval double 100  -- 200
```

"`#eval`로 실제로 계산할 수 있습니다. 함수처럼 입력을 넣으면 출력이 나옵니다."

---

### `theorem` — 참인 명제를 증명

```lean
theorem two_plus_two : 2 + 2 = 4 := rfl
```

"`theorem`은 '이게 참이다'를 주장하고 그 근거를 함께 제출하는 겁니다. 계산이 아니라 논리적 확인이 목적입니다."

---

**비유로 정리**

"`def`는 **레시피**입니다. 재료와 만드는 방법을 적어두는 거죠. 언제든 꺼내서 계산할 수 있습니다."

"`theorem`은 **식품 안전 인증서**입니다. '이 요리가 안전하다'는 것을 실제로 검사하고, 논리적으로 확인하고, 보증하는 문서죠."

---

**Q: def와 theorem 둘 다 `:=`를 쓰는데, 뭐가 다를까요?**

*(학생들이 생각할 시간 15초)*

"`:=` 오른쪽에 오는 것이 다릅니다."

| 구분 | `def` | `theorem` |
|------|-------|-----------|
| **목적** | 값/함수 정의 | 명제 증명 |
| **`:=` 오른쪽** | 값이나 계산식 | 증명 (전술들) |
| **결과 활용** | `#eval`로 계산 | 다른 증명에서 재사용 |

---

## 1.6 `#check`와 `#eval` — Lean에게 물어보기 (10분)

"Lean 4에는 증명과 별개로 '궁금한 것을 물어보는' 명령어가 있습니다. 이 두 명령어는 증명 코드에 포함되지 않습니다. 개발자가 탐구하고 디버깅할 때 쓰는 도구입니다."

---

### `#check` — "이것의 타입이 뭐야?"

*(코드를 하나씩 입력하며 결과를 보여준다.)*

```lean
#check Nat       -- Nat : Type
#check 42        -- 42 : Nat
#check "hello"   -- "hello" : String
#check 2 + 2 = 4 -- 2 + 2 = 4 : Prop
```

"타입을 확인하는 도구입니다. 42는 자연수(Nat) 타입, 'hello'는 문자열(String) 타입."

*(마지막 줄을 짚으며 강조)*

"그리고 `2 + 2 = 4`라는 수식 전체가 명제(Prop) 타입입니다. 이게 Lean의 핵심 철학입니다. **명제 자체가 하나의 타입입니다.** 그리고 그 타입의 '값'은 바로 증명이에요."

---

### `#eval` — "계산해서 결과 보여줘!"

```lean
#eval 2 + 3                      -- 5
#eval 10 * 20                    -- 200
#eval "Hello, " ++ "World!"      -- "Hello, World!"
#eval [1, 2, 3].length           -- 3
#eval [1, 2, 3].reverse          -- [3, 2, 1]
#eval (fun x => x * x) 7        -- 49
```

"실제로 계산해서 결과를 보여줍니다. `fun x => x * x`는 'x를 받아서 x * x를 돌려주는 함수'입니다. 거기에 7을 넣으면 49가 나옵니다."

---

### 중요한 함정

*(일부러 오류를 만들며)*

```lean
-- 이건 오류가 난다!
-- #eval 2 + 2 = 4
```

"왜 오류가 날까요?"

*(학생들이 생각할 시간 10초)*

"명제는 계산하는 게 아니라 **증명**하는 겁니다. `#eval`은 값을 계산하는 도구이므로 명제에는 쓸 수 없습니다. `2 + 2 = 4`는 데이터가 아니라 주장이니까요."

---

> **TIP — 시연 포인트**
>
> VS Code에서 `#eval 2 + 2 = 4`를 직접 입력해서 오류가 나는 모습을 보여준다.
> 오류 메시지를 학생들과 함께 읽는다.
> "오류 메시지는 적이 아닙니다. Lean이 '여기가 잘못됐어'라고 친절하게 알려주는 겁니다."
> 오류를 두려워하지 않는 태도를 심어주는 것이 이 시연의 목적이다.

---

### `#check` vs `#eval` 비교

| 명령어 | 묻는 것 | 예시 | 결과 |
|--------|---------|------|------|
| `#check` | 타입이 뭐야? | `#check 42` | `42 : Nat` |
| `#eval` | 값이 뭐야? | `#eval 2+3` | `5` |

## 1.7 Type과 Prop — Lean의 두 세계 (12분)

"Lean 4에는 크게 두 개의 세계가 공존합니다. 이걸 이해하면 코드를 읽는 눈이 달라집니다."

---

### Type의 세계 — 데이터와 계산

```lean
#check Nat       -- Nat : Type
#check Bool      -- Bool : Type
#check String    -- String : Type
#check List Nat  -- List Nat : Type
```

"Nat, Bool, String... 이것들은 데이터의 종류(Type)입니다. 이 세계에서 하는 일은 **계산**입니다."

```lean
#eval 2 + 3          -- 5
#eval true && false  -- false
#eval [1,2] ++ [3]   -- [1, 2, 3]
```

"계산하면 구체적인 값이 나옵니다."

---

### Prop의 세계 — 명제와 증명

```lean
#check 2 + 2 = 4         -- 2 + 2 = 4 : Prop
#check 1 = 2             -- 1 = 2 : Prop  (거짓이지만 명제는 맞음!)
#check ∀ n : Nat, n = n  -- ∀ (n : ℕ), n = n : Prop
```

"Prop의 세계에서는 참/거짓을 따질 수 있는 문장을 다룹니다. 그리고 이 세계에서 하는 일은 **증명**입니다."

---

*(여기서 학생들이 고개를 갸웃할 법한 부분을 먼저 짚는다.)*

"잠깐, `1 = 2`도 Prop입니까? 이건 거짓인데요."

"맞습니다. **거짓이어도 Prop입니다.** '1 = 2'는 참/거짓을 따질 수 있는 문장이니까요. 다만 거짓인 명제는 증명할 수가 없습니다. 증명을 시도하면 Lean이 영원히 빨간 줄을 긋습니다."

*(법정 비유)*

"법정 비유로 설명하겠습니다. 판사는 '이 사람이 범인이다'와 '이 사람이 범인이 아니다', 둘 다에 대해 판결을 내릴 수 있습니다. 판결 자체를 거부하지는 않아요. 다만 증거에 따라 결론이 달라질 뿐이죠. Prop도 마찬가지입니다. 참이든 거짓이든 명제는 명제입니다. 다만 참인 것만 **증명**할 수 있습니다."

---

```lean
-- 참인 명제 → 증명 가능
theorem t1 : 2 + 2 = 4 := rfl

-- 거짓인 명제 → 증명 불가능
-- theorem t2 : 1 = 2 := ???  -- 이건 영원히 증명 못 한다

-- 전칭 명제
theorem t3 : ∀ n : Nat, n = n := fun n => rfl
```

---

### 두 세계 비교표

| 특성 | Type (데이터 세계) | Prop (명제 세계) |
|------|------------------|----------------|
| 예시 | `Nat`, `Bool`, `String` | `2+2=4`, `∀n, n=n` |
| 값 | 구체적 데이터 | 증명(proof) |
| 확인 방법 | `#eval` (계산) | `theorem` (증명) |
| 비유 | 상자 안의 물건 | 판사의 판결문 |

"이 표를 외울 필요는 없습니다. 코드를 읽다가 '이건 데이터인가, 명제인가?'라는 감이 자연스럽게 생기기 시작하면 됩니다."

---

## 1.8 핵심 요약 복습 (5분)

"오늘 배운 것을 빠르게 정리합니다."

**Lean 4란?** 수학 증명을 검사하는 프로그램. 논리적 빈틈을 허용하지 않는다.

**수학의 3단계** — 정의 → 주장 → 증명. Lean에서는 `def`, `theorem`, 증명 전술로 대응한다.

**기억할 코드 패턴:**

```lean
-- 값/함수 정의
def 이름 : 타입 := 값
def 함수이름 (인자 : 타입) : 반환타입 := 본문

-- 정리 증명
theorem 정리이름 : 명제 := by
  전술들...

-- 탐문 도구
#check 표현식    -- 타입 확인
#eval 표현식     -- 값 계산
```

**`rfl`** — 계산하면 양쪽이 같은 경우. 변수가 있으면 안 된다.

**Type vs Prop** — 데이터 세계 vs 명제 세계. 각각 `#eval`과 `theorem`으로 다룬다.

### 문제 2: `#eval` 예측하기

</details>

### 문제 3: `rfl`로 증명하기

"sorry를 rfl로 바꾸면 됩니다. sorry는 '나중에 채울 빈칸'입니다. Lean이 일단 통과시켜주지만 노란 경고가 뜹니다. 저 자리를 실제 증명으로 채우세요."

```lean
theorem p1 : 3 + 5 = 8 := sorry
theorem p2 : 2 * 6 = 12 := sorry
theorem p3 : [1, 2].length = 2 := sorry
```

*(3분 후)*

<details>
<summary>정답 보기</summary>

```lean
theorem p1 : 3 + 5 = 8 := rfl
theorem p2 : 2 * 6 = 12 := rfl
theorem p3 : [1, 2].length = 2 := rfl
```

"sorry 자리에 rfl을 넣었을 때 노란 줄이 사라지고 InfoView에 `No goals`가 뜨면 성공입니다."
</details>

### 문제 4: `rfl`이 안 되는 이유 설명하기

"다음 코드는 왜 rfl로 증명되지 않을까요? 이유를 말로 설명해보세요."

```lean

```

*(2분 후)*

<details>
<summary>정답 보기</summary>

"n이 구체적인 숫자가 아니기 때문입니다. n = 0일 수도, n = 100일 수도 있어서 Lean은 `n + 0`을 구체적인 값으로 계산할 수 없습니다."

"단, 흥미로운 사실이 있습니다. Lean의 자연수 덧셈 정의에서 `n + 0 = n`은 정의 자체에 포함되어 있어서 실제로는 `rfl`로 되기도 합니다. 이런 경우를 **'정의에 의해 같다(definitionally equal)'**고 합니다. 지금은 이름만 알아두세요."
</details>

### 반드시 시연할 항목

1. `by` 뒤에 커서를 놓으면 InfoView에 초기 목표 `⊢ Even 4` 또는 `⊢ ∃ k, 4 = 2 * k`가 나타남을 보여준다.
2. `use 2` 앞뒤로 커서를 이동해서 목표가 `∃ k, 4 = 2 * k` → `4 = 2 * 2`로 바뀌는 것을 실시간으로 보여준다. **(이 수업의 핵심 시연)**
3. `rfl` 뒤에 커서를 놓으면 `No goals`가 표시됨을 보여준다.
4. `#eval 2 + 2 = 4`를 입력해서 오류가 나는 모습을 보여준다. 오류 메시지를 함께 읽는다.
5. `sorry`가 있을 때 노란 경고가 뜨는 것과 실제 증명으로 교체했을 때 사라지는 것을 보여준다.
