### 0. Lean 4 복습 — 10~20분

"지난 시간에 배운 것을 딱 세 줄로 복습합니다."

"Lean 4는 수학 증명을 컴퓨터가 검사해 주는 도구입니다. 틀리면 빨간 줄이 뜨고, 맞으면 빨간 줄이 사라집니다. 이 단순한 원리 하나가 전부입니다."

*(화면에 비교표를 보여주며)*

| 일상 | Lean 4 |
|------|--------|
| 수학 문제를 푼다 | 증명(proof)을 작성한다 |
| 선생님이 채점한다 | Lean이 타입 검사(type check)한다 |
| 식에서 값을 대입한다 | `rw`(치환)를 사용한다 |
| 정답입니다 | 에러 없음, No goals |
| 오답입니다 | 빨간 밑줄 + 에러 메시지 |

"이 표에서 오늘 수업과 가장 관련 깊은 줄이 보이시나요?"

"'식에서 값을 대입한다 → `rw`를 사용한다.' 오늘 수업의 절반이 이 한 줄에 담겨 있습니다."

### 1. 모든 것에는 타입이 있다 — 20~40분

"본론입니다. Lean 4의 세계에는 딱 하나의 규칙이 있습니다."

> **"모든 것에는 타입이 있다."**

"타입이란 '이 물건은 어떤 종류인가?'라는 질문의 답입니다."

*(화면에 표를 보여주며)*

| 물건 | 종류(타입) |
|------|-----------|
| 3 | 자연수 `ℕ` |
| -5 | 정수 `ℤ` |
| 3.14 | 실수 `ℝ` |
| `"안녕"` | 문자열 `String` |
| `true` | 불리언 `Bool` |

"이건 어렵지 않습니다. 우리가 이미 알고 있는 수의 분류를 그대로 씁니다."

---

#### 1.1 `#check` — "너 뭐야?"

```lean
#check 2 + 2        -- 2 + 2 : ℕ
#check (3.14 : ℝ)   -- 3.14 : ℝ
```

"`#check`는 '너 뭐야?'라고 묻는 명령어입니다. Lean은 '나는 ℕ(자연수)야'처럼 답합니다."

"재미있는 점은 `(3.14 : ℝ)` 이렇게 괄호 안에 콜론으로 타입을 직접 지정할 수도 있다는 겁니다. Lean에게 '이건 실수야'라고 알려주는 것이죠."

> **[ 시연 ]**
>
> VS Code에서 `#check 2 + 2`를 입력하고 InfoView에서 `2 + 2 : ℕ`이 나오는 것을 보여준다.
> 이어서 `#check (2 + 2 : ℝ)`을 입력하면 같은 `2 + 2`라도 타입이 `ℝ`로 바뀌는 것을 확인한다.
> "같은 식인데 타입이 다르게 붙을 수 있다는 것을 눈으로 확인한다."

---

#### 1.2 함수에도 타입이 있다

```lean
def f (x : ℕ) := x + 3

#check f      -- f : ℕ → ℕ
#check f 5    -- f 5 : ℕ
```

"함수도 타입을 가집니다. `f`는 자연수를 받아서 자연수를 돌려주는 함수이므로 타입이 `ℕ → ℕ`입니다."

"이 `→` 기호, 여기서는 '~를 받아서 ~를 돌려준다'는 함수의 타입을 나타냅니다. 잠깐요, 이 기호 나중에 논리에서도 나옵니다. 그때 이야기하겠습니다."

**Q: `#check f 5`의 결과가 `ℕ`인 이유가 뭘까요?**

*(학생들이 생각할 시간 10초)*

"`f 5`는 '함수 f에 5를 넣은 결과값'입니다. 결과값도 자연수이므로 타입이 `ℕ`입니다. `f 5`를 계산하면 5 + 3 = 8이지만, `#check`는 타입만 확인하므로 '이건 ℕ이야'라고만 답합니다."

---

#### 1.3 명제(Prop)도 타입이다 — 이것이 핵심

"자, 여기서 Lean 4의 가장 놀라운 아이디어가 나옵니다. 잘 들어주세요."

```lean
#check 2 + 2 = 4    -- 2 + 2 = 4 : Prop
```

"화면을 보시면, `2 + 2 = 4`라는 수식 전체가 `Prop`이라는 타입을 가집니다. `Prop`은 Proposition(명제)의 줄임말, 즉 '참이거나 거짓인 수학적 진술'입니다."

"이것이 왜 놀라울까요? 수학에서 명제는 판단의 대상이지 '물건'이 아니라고 생각해왔는데, Lean에서는 명제도 하나의 타입, 즉 물건의 종류로 취급한다는 겁니다."

*(잠깐 멈추며)*

"그렇다면 Prop 타입의 '값'은 뭘까요?"

**Q: 명제가 타입이라면, 그 타입의 값은 무엇일까요?**

*(학생들이 생각할 시간 15초)*

"정답은 **증명**입니다. '2 + 2 = 4'라는 명제 타입의 값은 '2 + 2 = 4의 증명'입니다."

*(비유를 들며)*

"빈 상자를 하나 상상해 보세요. 상자 뚜껑에 '2 + 2 = 4'라는 라벨이 붙어 있습니다. 이 상자 안에 넣을 수 있는 물건이 바로 증명입니다. Lean이 '이 상자에 뭔가 들어 있네, OK!'라고 확인해 주는 것이 바로 타입 검사입니다."

```lean
theorem easy : 2 + 2 = 4 := rfl

#check easy    -- easy : 2 + 2 = 4
```

"`theorem easy : 2 + 2 = 4 := rfl`을 보겠습니다. `easy`라는 이름의 증명을 만들었고, 그 타입은 `2 + 2 = 4`입니다. `#check easy`를 해보면 `easy : 2 + 2 = 4`가 나옵니다. easy라는 물건이 `2 + 2 = 4`라는 상자에 들어갔다는 것이죠."

"이 생각을 오늘 수업 내내, 아니 Lean을 배우는 내내 마음에 품고 있어야 합니다. **명제는 타입이고, 증명은 그 타입의 값이다.**"

---

#### 1.4 `sorry` — "아직 못 풀었어요"

```lean
theorem hard : ∀ x y z n : ℕ, n > 2 ∧ x * y * z ≠ 0 → x ^ n + y ^ n ≠ z ^ n := sorry
```

"이건 페르마의 마지막 정리입니다. 수학 역사에서 350년 동안 아무도 풀지 못한 문제입니다. 우리도 지금 당장은 못 풀죠. 그럴 때 쓰는 것이 `sorry`입니다."

"`sorry`는 시험에서 '풀이 생략'이라고 쓰는 것과 같습니다. Lean이 일단 통과시켜주지만 노란 경고를 표시합니다."

*(강조하며)*

"그런데 주의하세요. `sorry`가 있으면 Lean이 그 명제를 참이라고 **가정**해버립니다. 거짓인 명제도 `sorry`를 쓰면 '증명된' 것처럼 보입니다. 따라서 최종 증명에서는 `sorry`가 하나도 없어야 합니다."

> **[ 경고 ]**
>
> 연습 문제에서 `sorry`를 채워나가는 것은 좋은 전략이다.
> 코드 구조를 먼저 잡고, `sorry` 자리를 하나씩 실제 증명으로 교체해 나간다.
> 그러나 `sorry`가 포함된 코드는 신뢰할 수 없다. 최종 제출 전 반드시 `sorry`를 전부 제거해야 한다.

### 2. theorem vs lemma vs example — 40~60분

"두 번째 주제입니다. 처음 Lean 코드를 보면 `theorem`도 있고 `lemma`도 있어서 '뭐가 다른 거지?' 하게 됩니다."

"오늘 이 혼란을 완전히 해소해 드리겠습니다. 결론부터 드리면 이렇습니다."

> **"Lean 4에서 `theorem`과 `lemma`는 기능적으로 완전히 동일하다."**

*(학생들의 반응을 기다리며)*

"네, 같습니다. 코드로 확인해 봅시다."

```lean
theorem my_theorem : 2 + 3 = 5 := rfl
lemma my_lemma : 2 + 3 = 5 := rfl

#check my_theorem   -- my_theorem : 2 + 3 = 5
#check my_lemma     -- my_lemma   : 2 + 3 = 5
```

"완전히 동일합니다. Lean 내부적으로 같은 방식으로 처리됩니다. 둘 다 이름이 있고, 나중에 다른 증명에서 꺼내 쓸 수 있습니다."

---

**Q: 그렇다면 왜 두 가지 이름이 있을까요?**

*(학생들이 생각할 시간 20초)*

"인간을 위한 신호입니다. 코드를 읽는 사람에게 이 명제의 성격을 알려주는 것이죠."

"수학 교과서에서는 이렇게 구분합니다."

| 용어 | 의미 | 비유 |
|------|------|------|
| **정리**(theorem) | 중요하고 큰 결과 | 건물 전체 |
| **보조정리**(lemma) | 큰 정리를 돕는 작은 결과 | 건물의 벽돌 하나 |

"피타고라스 정리는 `theorem`이고, 그것을 증명하는 중간 단계의 명제들은 `lemma`입니다. 중요도와 역할의 차이죠."

"Lean에서도 그 관례를 따릅니다. `theorem`을 보면 '아, 이게 중요한 최종 결과구나', `lemma`를 보면 '아, 이건 보조 도구구나' 하고 직관적으로 알 수 있습니다."

"하지만 기계 입장에서는 구분 없이 같은 것입니다. 외울 것이 없어서 오히려 편하죠."

---

#### `example` — 이름 없는 연습 문제

```lean
example : 2 + 3 = 5 := rfl
```

"`example`은 이름이 없는 버전입니다. 연습 문제, 데모, 간단한 확인에 씁니다. 이름이 없으니 나중에 꺼내 쓸 수 없습니다."

"우리가 오늘 연습 문제를 풀 때 대부분 `example`로 씁니다."

*(표를 보여주며)*

| 키워드 | 이름 있음? | 재사용? | 용도 |
|--------|-----------|---------|------|
| `theorem` | O | O | 중요한 최종 결과 |
| `lemma` | O | O | 보조 도구, 중간 단계 |
| `example` | X | X | 연습, 데모, 예시 |

"이 표를 기억하세요. 코드를 읽을 때 키워드만 봐도 '이 명제가 어떤 역할을 하는구나'를 바로 알 수 있습니다."

## 2교시 — `rw` 치환(슈퍼포지션) 완전 정복 (60분)

"자, 2교시입니다. 오늘 수업에서 가장 중요한 시간입니다. `rw`를 완전히 이해하면 Lean 4 증명의 절반은 읽을 수 있습니다."

---

### 0~5분 — 도입: 치환이 뭔가?

"중학교 수학을 떠올려봅시다. `x = 3`일 때, `x + 2`는 얼마죠?"

*(학생들이 대답한다.)*

"맞습니다. 5입니다. `x` 자리에 `3`을 넣으면 되죠. 이게 치환(대입)입니다."

"Lean에서 `rw`가 정확히 이 일을 합니다. 등식 `A = B`를 알고 있을 때, 어떤 식에서 `A`가 보이면 `B`로 바꿔치기를 합니다."

"이것을 수학 논리학 용어로 **슈퍼포지션(superposition)**이라고도 합니다. '같다고 알려진 것을 겹쳐놓기'라는 뜻이죠."

---

### 5~25분 — `rw` 기초: 단계별로 따라가기

"가장 간단한 예제부터 시작합니다. 코드를 입력하면서 InfoView를 함께 봅니다."

```lean
example (a b c : ℝ) : a * b * c = b * (a * c) := by
  rw [mul_comm a b]      -- a * b를 b * a로 바꾼다
  rw [mul_assoc b a c]   -- b * a * c를 b * (a * c)로 바꾼다
```

"이 코드를 보면 증명해야 할 것이 `a * b * c = b * (a * c)`입니다. 등호 왼쪽과 오른쪽의 모양이 다릅니다. `rw`로 왼쪽을 오른쪽과 같은 모양으로 만들어가는 것이 목표입니다."

"한 단계씩 따라가겠습니다."

*(VS Code에서 코드를 입력하며 커서를 이동한다.)*

**시작 상태 — `by` 뒤에 커서**

```
⊢ a * b * c = b * (a * c)
```

"`⊢` 아래가 지금 증명해야 할 목표입니다. 왼쪽의 `a * b * c`와 오른쪽의 `b * (a * c)`가 다릅니다."

---

**1단계 — `rw [mul_comm a b]` 뒤에 커서**

```
⊢ b * a * c = b * (a * c)
```

"`mul_comm a b`는 'a * b = b * a'라는 곱셈의 교환법칙입니다. 이 사실을 이용해서 목표 안의 `a * b`를 `b * a`로 바꿨습니다. 왼쪽이 `b * a * c`로 바뀐 것이 보이시나요?"

"비유하자면, 종이에 적힌 `a * b`라는 글자 위에 `b * a`라는 스티커를 붙인 겁니다. 스티커가 붙으면 원래 글자 대신 스티커 내용이 보입니다."

---

**2단계 — `rw [mul_assoc b a c]` 뒤에 커서**

```
⊢ b * (a * c) = b * (a * c)
```

"`mul_assoc b a c`는 'b * a * c = b * (a * c)'라는 결합법칙입니다. `b * a * c`를 `b * (a * c)`로 바꿨습니다. 이제 양쪽이 같아졌습니다."

"양쪽이 같아지는 순간 Lean이 자동으로 '증명 완료'라고 판정합니다. InfoView에 `No goals`가 나타납니다."

> **[ 반드시 시연 ]**
>
> VS Code에서 직접 코드를 입력하고 커서를 `by` 뒤 → 첫 번째 `rw` 뒤 → 두 번째 `rw` 뒤로 순서대로 이동하면서 InfoView 변화를 학생들에게 보여준다.
> "이 커서 이동이 Lean 4 학습에서 가장 강력한 도구입니다. 탐정이 증거를 하나씩 따라가듯 증명의 흐름을 추적할 수 있습니다."
> 학생들도 직접 따라 입력하게 한다. (5분 소요)

---

**Q: `⊢` 기호가 보이지 않는 순간이 있었는데, 그건 무슨 뜻인가요?**

"좋은 질문입니다. `⊢`는 아직 증명해야 할 목표가 있을 때만 나타납니다. 증명이 완료되면 더 이상 증명할 것이 없으므로 기호도 사라지고 `No goals`가 뜹니다."

"중간 과정을 다시 보고 싶다면 커서를 코드 위로 이동하면 됩니다."

```
1. by 바로 뒤를 클릭   →  ⊢ a * b * c = b * (a * c)  (시작 목표)
2. 첫 번째 rw 줄 끝   →  ⊢ b * a * c = b * (a * c)  (1단계 후)
3. 두 번째 rw 줄 끝   →  No goals                   (증명 완료)
```

---

### 25~40분 — `rw`의 방향과 활용

#### 역방향 치환: `rw [← h]`

"`rw`는 기본적으로 등식의 왼쪽을 오른쪽으로 바꿉니다. 그런데 반대 방향으로 바꾸고 싶을 때가 있습니다."

```
mul_comm a b  :  a * b = b * a
                 ─────   ─────
                 왼쪽     오른쪽

rw [mul_comm a b]    →  목표에서 "a * b"를 "b * a"로 바꿈  (정방향)
rw [← mul_comm a b]  →  목표에서 "b * a"를 "a * b"로 바꿈  (역방향)
```

"← 기호는 VS Code에서 `\l` 또는 `\leftarrow`를 입력하면 됩니다."

*(잠깐 웃으며)*

"처음 이걸 배우면 언제 정방향을 쓰고 언제 역방향을 쓰는지 헷갈립니다. 정답은 '지금 목표에 어떤 패턴이 있는지 보고 결정한다'입니다. 직접 해보다 보면 감이 생깁니다."

---

#### 여러 치환을 한 줄에

```lean
-- 긴 버전 (한 줄씩)
example (a b c : ℝ) : a * b * c = b * (a * c) := by
  rw [mul_comm a b]
  rw [mul_assoc b a c]

-- 짧은 버전 (한 줄에)
example (a b c : ℝ) : a * b * c = b * (a * c) := by
  rw [mul_comm a b, mul_assoc b a c]
```

"둘은 완전히 같은 증명입니다. 여러 번 치환할 때 콤마로 이어서 쓸 수 있습니다."

---

#### `rw`의 재료: `#check`로 확인하기

"`rw`에 넣을 수 있는 재료는 `A = B` 또는 `A ↔ B` 형태의 정리입니다."

```lean
#check (mul_comm : ∀ a b : ℝ, a * b = b * a)
#check (mul_assoc : ∀ a b c : ℝ, a * b * c = a * (b * c))
#check (add_comm : ∀ a b : ℝ, a + b = b + a)
#check (two_mul : ∀ a : ℝ, 2 * a = a + a)
```

"정리 이름이 기억나지 않을 때는 `#check`로 확인합니다. 어떤 정리가 어떤 등식인지 바로 볼 수 있습니다."

> **[ 시연 ]**
>
> VS Code에서 `import Mathlib.Tactic` 이후 `#check mul_comm`을 입력한다.
> `mul_comm : ∀ (a b : ?α), a * b = b * a`가 나타나는 것을 보여준다.
> 이어서 `#check add_comm`을 입력해서 익숙한 덧셈 교환법칙을 확인한다.
> "모르는 정리 이름이 나오면 #check를 먼저 써보는 것이 좋습니다. 그 정리가 어떤 형태인지 바로 파악할 수 있습니다."

### 40~55분 — 가설에서 치환: `rw [...] at h`

"지금까지는 목표(goal)에서 치환을 했습니다. 그런데 가설(hypothesis), 즉 이미 알고 있는 사실에서도 치환할 수 있습니다."

"이게 처음엔 이상하게 보입니다. '가설을 왜 바꿔?'라고 생각할 수 있습니다. 하지만 실제 수학 풀이에서는 주어진 조건을 변형해 가면서 목표에 맞추는 작업이 많습니다."

```lean
example (a b c d : ℝ) (hyp : c = d * a + b) (hyp' : b = a * d) : c = 2 * a * d := by
  rw [hyp'] at hyp          -- hyp에서 b를 a * d로 바꿈
                              -- hyp: c = d * a + a * d
  rw [mul_comm d a] at hyp  -- hyp에서 d * a를 a * d로 바꿈
                              -- hyp: c = a * d + a * d
  rw [← two_mul (a * d)] at hyp  -- hyp에서 a*d + a*d를 2*(a*d)로 바꿈
  rw [← mul_assoc 2 a d] at hyp  -- hyp: c = 2 * a * d
  exact hyp                       -- 목표와 hyp가 정확히 같으므로 끝!
```

"이 코드를 단계별로 따라가겠습니다. 핵심은 `hyp`라는 가설이 단계마다 어떻게 변해가는지 보는 것입니다."

*(커서를 이동하며 각 단계에서 InfoView를 보여준다.)*

```
시작:
  hyp  : c = d * a + b
  hyp' : b = a * d
  ⊢ c = 2 * a * d

rw [hyp'] at hyp 후:
  hyp  : c = d * a + a * d    ← b가 a*d로 바뀌었다!
  hyp' : b = a * d
  ⊢ c = 2 * a * d

rw [mul_comm d a] at hyp 후:
  hyp  : c = a * d + a * d    ← d*a가 a*d로 바뀌었다!
  ⊢ c = 2 * a * d

rw [← two_mul (a * d)] at hyp 후:
  hyp  : c = 2 * (a * d)
  ⊢ c = 2 * a * d

rw [← mul_assoc 2 a d] at hyp 후:
  hyp  : c = 2 * a * d        ← 목표와 일치!
  ⊢ c = 2 * a * d

exact hyp 후:
  No goals                    ← 증명 완료!
```

"마지막의 `exact hyp`는 '`hyp`가 목표와 정확히 같으므로, 이것으로 증명이 끝난다'는 전술입니다."

"가설을 변형해서 목표와 같은 모양으로 만들고 `exact`로 끝내는 패턴, 이게 Lean 4 증명의 가장 기본적인 흐름입니다."

---

### 55~60분 — `ring`: 한 줄로 끝내기

"자, 솔직히 말씀드리겠습니다. 방금 그 네 단계짜리 치환, 실제로는 이렇게 끝납니다."

```lean
import Mathlib.Tactic

example (a b c : ℝ) : a * b * c = b * (a * c) := by
  ring
```

*(잠깐 멈추며)*

"네. `ring` 한 줄이면 됩니다."

"대수적인 등식 — 교환법칙, 결합법칙, 분배법칙이 얽힌 어떤 복잡한 등식도 `ring`이 자동으로 증명합니다."

"그렇다면 왜 방금 `rw`를 그렇게 공들여 배웠을까요?"

**Q: `rw`를 배운 이유가 뭘까요?**

*(학생들의 대답을 듣는다.)*

"`ring`이 통하지 않는 경우가 있기 때문입니다. 부등식에는 `ring`을 쓸 수 없습니다. 가설을 활용해야 하는 경우에도 `ring`만으로는 부족합니다. 그리고 수학적 논리가 어떻게 흘러가는지 이해하려면 단계별 `rw`가 필수입니다."

"`ring`은 강력한 자동화 도구입니다. 그러나 자동화 도구를 믿고 쓰려면 그 도구가 무엇을 하는지 알아야 합니다. 그게 `rw`를 배운 이유입니다."

| 방법 | 줄 수 | 쓰는 때 |
|------|-------|---------|
| `rw` 단계별 | 여러 줄 | 각 단계를 직접 이해하고 싶을 때 |
| `ring` 자동 | 1줄 | 대수 등식임이 분명할 때 |

> **[ 경고 ]**
>
> `ring`은 **등식(=)만** 해결한다.
> 부등식(`≤`, `<`) 또는 논리 명제에는 `ring`을 쓸 수 없다.
> 부등식은 `linarith`, 자연수 산술은 `omega`, 구체적 수치는 `norm_num`을 쓴다.

---

### `rw` 전체 요약표

| 표현 | 의미 |
|------|------|
| `rw [h]` | 목표에서 h의 왼쪽 → 오른쪽으로 치환 |
| `rw [← h]` | 목표에서 h의 오른쪽 → 왼쪽으로 치환 |
| `rw [h] at hyp` | 가설 `hyp`에서 치환 |
| `rw [h1, h2, h3]` | 연속 치환 (왼쪽부터 순서대로) |

"`rw`에 넣는 것은 반드시 `A = B` 또는 `A ↔ B` 형태여야 합니다. `A → B` 형태는 `rw`에 쓸 수 없습니다. 이 제한이 나중에 오류의 원인이 되는 경우가 많으니 기억해 두세요."

## 3교시 — `→`와 `↔`, `calc`, 연습 문제 (60분)

---

### 0~25분 — `→`(함의)와 `↔`(동치)의 차이

"오늘 수업의 마지막 개념적 핵심입니다. `→`와 `↔`. 수학에서 가장 많이 혼동하는 두 기호입니다."

"일상 비유로 시작합니다."

| 기호 | 읽는 법 | 뜻 | 일상 비유 |
|------|---------|----|-----------| 
| `→` | "이면" (if...then) | 한 방향 | 비가 오면 → 땅이 젖는다 |
| `↔` | "동치" (iff) | 양 방향 | 정삼각형 ↔ 세 변의 길이가 같다 |

---

#### `→` — 한 방향 화살표

"A → B는 'A이면 B이다'입니다."

"예: '비가 오면 땅이 젖는다.'"

"비가 올 때(A가 참) → 땅이 젖는다(B가 참). 여기까지는 당연합니다."

"그런데 반대 방향은 어떤가요? '땅이 젖었으면 비가 온 것이다.' 이게 참일까요?"

*(학생들의 반응을 기다린다.)*

"틀릴 수 있습니다. 스프링클러 때문에 젖었을 수도 있으니까요."

**핵심 포인트**: A → B가 참이어도 B → A가 참인 건 아닙니다.

"이게 `→`의 본질입니다. 한쪽 방향만 보장합니다."

---

#### `↔` — 양 방향 화살표

"A ↔ B는 'A이면 B이고, B이면 A이다'입니다. 두 방향이 모두 성립합니다."

"예: '정삼각형이다 ↔ 세 변의 길이가 같다.'"

"정삼각형이면 세 변이 같고, 세 변이 같으면 정삼각형입니다. 어느 방향이든 성립합니다."

"수학에서 A ↔ B를 보면 'A와 B는 완전히 동등한 조건이다'라는 뜻입니다. 가장 강력한 형태의 관계죠."

---

#### Lean 4에서 `→`의 두 가지 역할

```lean
-- 역할 1: 함수의 타입
def f : ℕ → ℕ := fun n => n + 3

-- 역할 2: 논리적 함의
#check ∀ x : ℝ, 0 ≤ x → |x| = x
```

"Lean에서 `→`는 함수 타입을 나타낼 때도 쓰고 논리적 함의를 나타낼 때도 씁니다. 이 둘이 왜 같은 기호를 쓸까요?"

"이게 Lean의 철학입니다. 'P의 증명을 받아서 Q의 증명을 돌려주는 함수'가 바로 'P이면 Q이다'의 증명이기 때문입니다. 이것을 **커리-하워드 대응**(Curry-Howard correspondence)이라 합니다. 오늘은 이름만 알아두세요."

---

#### `→` 증명하기: `intro`

"`P → Q`를 증명하는 절차는 단순합니다."

"1단계: 'P가 참이라고 가정하자'고 선언한다. Lean에서는 `intro h`입니다."

"2단계: 그 가정 아래서 Q를 증명한다."

```lean
example (a b c : ℝ) : a ≤ b → a + c ≤ b + c := by
  intro h          -- h : a ≤ b  라고 가정
  linarith         -- 산술 자동 증명기가 해결
```

```
[ InfoView — intro h 전후 ]

intro h 전:   ⊢ a ≤ b → a + c ≤ b + c

intro h 후:   h : a ≤ b
              ⊢ a + c ≤ b + c
              (화살표가 사라지고 h가 가정으로 추가됨!)
```

"`intro h`를 쓰는 순간 → 기호가 사라지고, 화살표 왼쪽의 `a ≤ b`가 가정 `h`로 변환됩니다. 그리고 남은 목표는 `a + c ≤ b + c`가 됩니다."

"이제 `h : a ≤ b`라는 재료가 생겼으니 `linarith`가 자동으로 증명합니다."

---

#### Lean 4에서 `↔` 다루기

```lean
-- exp_le_exp : exp a ≤ exp b ↔ a ≤ b
#check (exp_le_exp : exp a ≤ exp b ↔ a ≤ b)
```

"`↔`는 두 개의 `→`로 이루어져 있습니다. 각각 꺼낼 수 있습니다."

| 표현 | 의미 | 방향 |
|------|------|------|
| `h.mp` | A → B | 정방향 |
| `h.mpr` | B → A | 역방향 |
| `h.1` | A → B | 정방향 (`h.mp`와 같음) |
| `h.2` | B → A | 역방향 (`h.mpr`와 같음) |

"mp는 modus ponens(긍정 논법)의 줄임말입니다. mpr은 그 역방향입니다."

---

#### `↔` 정리를 `rw`에 쓰기 — 핵심!

"여기서 중요한 사실이 나옵니다. `rw`는 `=` 등식뿐 아니라 `↔` 동치에도 쓸 수 있습니다."

```lean
-- h : a ≤ b
example (a b : ℝ) (h : a ≤ b) : exp a ≤ exp b := by
  rw [exp_le_exp]    -- 목표 exp a ≤ exp b를 동치인 a ≤ b로 바꿈
  exact h            -- h가 정확히 a ≤ b이므로 끝!
```

```
[ InfoView — rw [exp_le_exp] 전후 ]

rw 전:   ⊢ exp a ≤ exp b

rw 후:   ⊢ a ≤ b          ← 동치인 다른 명제로 교체됨!
```

"`rw [exp_le_exp]`를 쓰는 순간 목표 `exp a ≤ exp b`가 동치인 `a ≤ b`로 바뀝니다. 그리고 `h : a ≤ b`가 이미 있으므로 `exact h`로 끝납니다."

"또 다른 방법도 있습니다."

```lean
-- 방법 2: .mpr 직접 사용
example (a b : ℝ) (h : a ≤ b) : exp a ≤ exp b := by
  exact exp_le_exp.mpr h
```

"`exp_le_exp.mpr`은 '역방향: `a ≤ b → exp a ≤ exp b`'입니다. `h`(a ≤ b의 증거)를 넣으면 바로 증명됩니다."

**Q: 방법 1과 방법 2 중 어느 것이 더 직관적인가요?**

*(학생들에게 물어본다.)*

"둘 다 맞습니다. 상황에 따라 더 읽기 편한 것을 쓰면 됩니다. 방법 1(`rw`)은 '목표를 바꿔가며 해결하는' 느낌, 방법 2(`.mpr`)는 '이미 있는 정리를 바로 적용하는' 느낌입니다."

---

#### `↔` 증명하기: `constructor`

"`A ↔ B`를 직접 증명할 때는 두 방향을 각각 증명해야 합니다. `constructor`로 목표를 쪼갭니다."

```lean
example (a b : ℝ) (h : a ≤ b) : ¬(b ≤ a) ↔ a ≠ b := by
  constructor
  · -- 정방향: ¬(b ≤ a) → a ≠ b
    intro h₁ h₂
    apply h₁
    rw [h₂]
  · -- 역방향: a ≠ b → ¬(b ≤ a)
    intro h₁ h₂
    apply h₁
    exact le_antisymm h h₂
```

"`constructor`를 쓰면 `↔`가 두 개의 `→` 목표로 쪼개집니다. `·` (가운뎃점)으로 각 목표에 집중하여 증명합니다."

---

#### `→`와 `↔` 비교 요약

| 특성 | `→` (함의) | `↔` (동치) |
|------|-----------|-----------|
| 방향 | 한 방향 | 양 방향 |
| 분해 | 불가 | `.mp`(정방향) + `.mpr`(역방향) |
| `rw`에 쓸 수 있나? | 불가 | 가능 |
| 증명 방법 | `intro`로 전제 도입 | `constructor`로 두 방향 각각 |

"`→`는 `rw`에 쓸 수 없습니다. `rw`는 등호(=)와 동치(↔)만 처리합니다. 이 사실을 모르면 `→` 정리를 `rw`에 넣다가 계속 오류가 납니다."

---

### 25~40분 — `calc` — 단계별 풀이

"수학 시험에서 풀이 과정을 단계별로 쓰듯이, Lean에서도 `calc`로 단계별 계산을 보여줄 수 있습니다."

```lean
example (a b : ℝ) : (a + b) * (a + b) = a * a + 2 * (a * b) + b * b := by
  calc
    (a + b) * (a + b)
        = a * a + b * a + (a * b + b * b) := by rw [mul_add, add_mul, add_mul]
      _ = a * a + (b * a + a * b) + b * b := by rw [← add_assoc, add_assoc (a * a)]
      _ = a * a + 2 * (a * b) + b * b     := by rw [mul_comm b a, ← two_mul]
```

"각 줄의 `_`는 '바로 윗줄의 결과'를 의미합니다. 마치 수학 교과서에서 이렇게 쓰는 것처럼요."

```
(a + b)(a + b)
= a² + ba + ab + b²      ← 분배법칙
= a² + (ba + ab) + b²    ← 결합법칙 조정
= a² + 2ab + b²          ← ba = ab이므로 ba + ab = 2ab
```

"이것을 그대로 Lean 코드로 옮긴 겁니다. 각 등호 뒤의 `by rw [...]`가 그 단계가 성립하는 이유를 설명합니다."

---

#### 사실 `ring`으로도 됩니다

```lean
example (a b : ℝ) : (a + b) * (a + b) = a * a + 2 * (a * b) + b * b := by
  ring
```

*(웃으며)*

"`ring` 한 줄이면 됩니다. 그렇지만 `calc`는 논리의 흐름을 보여주는 데 가치가 있습니다. 등호와 부등호가 섞인 복잡한 추론에서 `calc`는 각 단계가 명확하게 드러나서 읽기 좋습니다."

#### 연습 1: `rw` 기초 (괄호 채우기)

"이 문제는 `{?}` 자리에 알맞은 정리 이름을 채우는 겁니다."

```lean
example (a b c : ℝ) : c * b * a = b * (a * c) := by
  rw [{?} c b]     -- c * b를 b * c로 바꾸는 정리
  rw [{?} b c a]   -- b * c * a를 b * (c * a)로 바꾸는 정리
  rw [{?} c a]     -- c * a를 a * c로 바꾸는 정리
```

"힌트: 교환법칙은 `mul_comm`, 결합법칙은 `mul_assoc`입니다. `#check mul_comm`으로 정확한 형태를 확인해 보세요."

*(4분 후)*

<details>
<summary>정답 보기</summary>

```lean
example (a b c : ℝ) : c * b * a = b * (a * c) := by
  rw [mul_comm c b]    -- c * b → b * c  (목표: b * c * a = b * (a * c))
  rw [mul_assoc b c a] -- b * c * a → b * (c * a)  (목표: b * (c * a) = b * (a * c))
  rw [mul_comm c a]    -- c * a → a * c  (목표: b * (a * c) = b * (a * c))
```

"단계별 설명: c * b를 교환법칙으로 b * c로, b * c * a를 결합법칙으로 b * (c * a)로, c * a를 교환법칙으로 a * c로 바꾸면 양쪽이 같아집니다."
</details>

#### 연습 2: `rw` — `ring` 없이

```lean
example (a b : ℝ) : (a + b) * (a - b) = a ^ 2 - b ^ 2 := by
  rw [pow_two]     -- 힌트: a^2 = a * a 로 바꾸는 정리
  sorry            -- 나머지는 직접 해보자!
```

*(3분 후)*

<details>
<summary>정답 보기</summary>

```lean
-- ring을 쓰면 한 줄:
example (a b : ℝ) : (a + b) * (a - b) = a ^ 2 - b ^ 2 := by
  ring
```

"`rw`만으로 하려면 꽤 길어집니다. 이 단계에서는 `ring`이 현명한 선택입니다. `ring`은 이런 대수 항등식에 최적화되어 있습니다."
</details>

#### 연습 3: `→`의 이해 (sorry 채우기)

```lean
-- "a ≤ b이고 c < d이면, a + c < b + d이다"
example (a b c d : ℝ) (h₁ : a ≤ b) (h₂ : c < d) : a + c < b + d := by
  sorry
  -- 힌트 1: add_lt_add_of_le_of_lt라는 정리가 있다
  -- 힌트 2: linarith로도 한 방에 해결된다
```

"두 가지 힌트를 드렸습니다. 먼저 `linarith`로 시도해 보세요. 그다음 `add_lt_add_of_le_of_lt`를 `#check`로 확인하고 `exact`로 써보세요."

*(4분 후)*

<details>
<summary>정답 보기</summary>

```lean
-- 방법 1: linarith 자동 증명
example (a b c d : ℝ) (h₁ : a ≤ b) (h₂ : c < d) : a + c < b + d := by
  linarith

-- 방법 2: 정리 직접 적용
example (a b c d : ℝ) (h₁ : a ≤ b) (h₂ : c < d) : a + c < b + d := by
  exact add_lt_add_of_le_of_lt h₁ h₂
```

"`add_lt_add_of_le_of_lt`는 'a ≤ b이고 c < d이면 a + c < b + d'라는 정리입니다. h₁과 h₂를 넣으면 바로 증명됩니다. `linarith`는 이런 선형 부등식을 자동으로 처리합니다."
</details>

#### 연습 4: `↔`의 이해 (sorry 채우기)

```lean
-- exp_le_exp : exp a ≤ exp b ↔ a ≤ b 를 이용
example (a b : ℝ) (h : a ≤ b) : exp a ≤ exp b := by
  sorry
  -- 힌트 1: rw [exp_le_exp] 를 쓰거나
  -- 힌트 2: exact exp_le_exp.mpr h 를 쓰거나
```

*(4분 후)*

<details>
<summary>정답 보기</summary>

```lean
-- 방법 1: rw로 목표를 동치인 것으로 변환
example (a b : ℝ) (h : a ≤ b) : exp a ≤ exp b := by
  rw [exp_le_exp]
  exact h

-- 방법 2: .mpr로 역방향 직접 적용
example (a b : ℝ) (h : a ≤ b) : exp a ≤ exp b := by
  exact exp_le_exp.mpr h

-- 방법 3: 한 줄 증명항 (term mode)
example (a b : ℝ) (h : a ≤ b) : exp a ≤ exp b :=
  exp_le_exp.mpr h
```

"방법 1: `rw [exp_le_exp]`가 목표 `exp a ≤ exp b`를 동치인 `a ≤ b`로 바꿔줍니다. 그러면 h가 정확히 그 목표이므로 `exact h`로 끝납니다."

"방법 2: `.mpr`은 '역방향: a ≤ b → exp a ≤ exp b'를 꺼냅니다. 거기에 h를 넣으면 바로 증명됩니다."
</details>

---

#### 연습 5: `rw at` (가설에서 치환)

```lean
example (a b c d : ℝ) (hyp : c = b * a - d) (hyp' : d = a * b) : c = 0 := by
  sorry
  -- 힌트: hyp'를 hyp에 대입하면 c = b * a - a * b
  --       b * a = a * b이므로 c = a * b - a * b = 0
  -- 사용 가능한 정리: mul_comm, sub_self
```

*(5분 후)*

<details>
<summary>정답 보기</summary>

```lean
example (a b c d : ℝ) (hyp : c = b * a - d) (hyp' : d = a * b) : c = 0 := by
  rw [hyp'] at hyp          -- hyp: c = b * a - a * b
  rw [mul_comm b a] at hyp  -- hyp: c = a * b - a * b
  rw [sub_self] at hyp      -- hyp: c = 0
  exact hyp
```

"단계별 설명:"

"1. `rw [hyp'] at hyp`: hyp에서 d를 a*b로 치환 → `hyp : c = b * a - a * b`"

"2. `rw [mul_comm b a] at hyp`: b*a를 a*b로 치환 → `hyp : c = a * b - a * b`"

"3. `rw [sub_self] at hyp`: `a * b - a * b = 0`이므로 → `hyp : c = 0`"

"4. `exact hyp`: 목표 `c = 0`과 일치하므로 증명 완료."
</details>

### 마무리 — 전체 요약 (5분)

"오늘 배운 핵심 개념을 정리합니다."

---

**Lean 4의 핵심 원리 — 타입 계층**

```
모든 것에는 타입이 있다
├── 숫자 3의 타입: ℕ (자연수)
├── 함수 f의 타입: ℕ → ℕ
├── 명제 2+2=4의 타입: Prop
└── 증명 rfl의 타입: 2+2=4  (명제 그 자체가 타입!)
```

---

**theorem vs lemma vs example**

```
theorem (정리)   ─── 이름 O, 재사용 O, "중요한 최종 결과"
lemma (보조정리) ─── 이름 O, 재사용 O, "보조 도구"
example (예시)  ─── 이름 X, 재사용 X, "연습용 / 데모용"

* Lean 4에서 theorem과 lemma는 기능적으로 완전히 동일하다.
```

---

**`rw` 치환(슈퍼포지션) 요약**

```
rw [h]          목표에서 h의 왼쪽 → 오른쪽으로 치환
rw [← h]        역방향 치환
rw [h] at hyp   가설 hyp에서 치환
rw [h1, h2]     연속 치환

* h는 반드시 "A = B" 또는 "A ↔ B" 형태여야 한다.
* → 형태는 rw에 쓸 수 없다.
```

---

**`→`(함의)와 `↔`(동치)**

```
→ (한 방향)
  ├── 증명: intro로 전제 도입 후 결론 증명
  └── rw에 쓸 수 없다

↔ (양 방향 = → + ←)
  ├── .mp: 정방향  /.mpr: 역방향
  ├── 증명: constructor로 두 방향 각각
  └── rw에 쓸 수 있다!
```

---

**지금까지 배운 전술 목록**

| 전술 | 용도 |
|------|------|
| `rw [h]` | 치환 (바꿔치기) |
| `exact h` | "이것이 정확히 증명이다" |
| `intro h` | 가정 도입 ("~라고 하자") |
| `apply h` | 정리 역방향 적용 |
| `constructor` | `∧` 또는 `↔`를 두 목표로 분리 |
| `ring` | 환의 산술 등식 자동 증명 |
| `linarith` | 선형 부등식 자동 증명 |
| `sorry` | 미완성 표시 (나중에 채울 것!) |

### 반드시 시연할 항목

1. **`#check` 타입 확인**: `#check 2 + 2`, `#check (2 + 2 : ℝ)` 비교. 같은 식도 타입이 다를 수 있음을 확인.
2. **`rw` 단계별 InfoView**: `example (a b c : ℝ) : a * b * c = b * (a * c)`에서 `by` 뒤 → 첫 번째 `rw` 뒤 → 두 번째 `rw` 뒤 커서 이동. 목표 변화 실시간 확인. **(2교시 핵심 시연)**
3. **`rw at hyp` InfoView**: `hyp`가 단계마다 어떻게 변해가는지 추적.
4. **`ring` vs `rw` 비교**: `rw` 여러 줄 후 `ring` 한 줄로 같은 결과를 보여주며 카타르시스 제공.
5. **`intro h` InfoView**: `→` 목표에 `intro h`를 쓰면 화살표가 사라지고 가정이 생기는 것을 실시간으로 확인.
6. **`rw [↔ 정리]`**: `exp_le_exp`를 `rw`에 넣었을 때 목표가 바뀌는 것 확인.
